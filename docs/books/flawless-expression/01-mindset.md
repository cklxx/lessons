# 心智模型：正确性的定义与边界

绝对正确的表达不是追求形式上的完美，而是让信息在目标场景下被**无歧义理解、无偏差执行、可复现验证**。本章用三个心智模型帮你判断何时需要追求极高正确性，并给出一套可直接套用的“正确性分级”工作流与模板。

## 你将收获什么
- 一套判断“这次沟通要做到多正确”的三步分级法（含门槛与例外）。
- 一个可复制的“正确性分级卡片”模板：场景、后果、门槛、证据、回滚一次写清。
- 一份最低门槛检查清单：交付前快速扫掉高频歧义与隐含假设。
- 三类典型失败样本：隐含假设、模糊量词、概念漂移（含复现与修复）。

## 正确性的三维度
事实、逻辑与语用是并行的三根支柱：

- **事实正确**：信息源可靠、数据可追溯。任何数字、引语、日期都能找到原始出处且可独立验证。
- **逻辑正确**：前提、推论与结论链条自洽。推理步骤应清晰可审计，结论与前提一一对应，无逻辑跳跃。
- **语用正确**：表达符合场景与受众。避免让读者在缺失上下文信息的情况下自行补全（例如对非技术读者使用未解释的缩写）。

## 成本模型：何时值得追求绝对正确
并非所有内容都需要全量校验，应根据“后果 + 耦合”决定投入：

- **高后果场景**：法律条款、合规声明、金融报价、医疗或安全指令。必须严谨，容错率极低。
- **高耦合场景**：跨团队协作或自动化执行环节。错误会级联放大，值得额外显式化。
- **机器执行场景**：当内容将被程序或代理执行时，必须提供参数化、可复现的描述，并预留回滚入口。

## 关键工作流（可直接套用）：正确性分级三步法
在开始写作或沟通前，用三步确定你要做到的正确性等级与门槛。

**Step 1：评估后果（Impact）**
- 问题：如果错了，会发生什么？只是返工，还是资损/泄露/事故？
- 门槛：涉及资金/权限/安全/合规的一律按“审计级”处理（默认最高等级）。

**Step 2：检查耦合（Coupling）**
- 问题：这段信息会成为下游输入吗？下游是人（可纠偏）还是系统/脚本/Agent（容错低）？
- 门槛：下游是机器或跨团队陌生协作时，必须消除模糊量词与隐含假设。

**Step 3：定义证据链（Evidence）**
- 问题：本次结论要用什么证据支撑？是“凭经验”，还是“可复核数据/日志/可复跑步骤”？
- 行动：填下面的“正确性分级卡片”，把门槛、证据与回滚在交付前写死。

例外策略（允许降级的情况）：只影响个人效率、可随时撤回、不进入下游流程、且失败成本低于一次重做的场景。

## 示例（可复制）：正确性分级卡片模板
> 用法：涉及关键决策、跨团队协作或机器执行的内容，先填这张卡，再开始写正文。

```markdown
| 维度 | 内容填写 |
| :--- | :--- |
| 场景 | <例如：支付网关接口变更公告> |
| 后果等级 | <例如：Critical - 错误会导致资损或对账异常> |
| 最低验证门槛 | <例如：预发通过回归 + 核心链路对账抽检> |
| 证据要求 | <例如：对比表/日志/截图/计算过程/链接> |
| 失败判定 | <例如：主指标未达标或守门指标退化即失败> |
| 回滚方案 | <例如：一键关闭开关，回滚耗时 < 10 分钟> |
```

## 检查清单（最低门槛）
在提交任何正式文档/公告/交付说明前，至少完成以下自检：

1. 所有数字是否注明时间、单位、来源与适用范围？
2. 是否清除了“尽快/大概/适当/多数”等不可衡量词（或已给阈值与窗口）？
3. 是否写清了前置条件（权限/环境/依赖/数据版本）？
4. 核心术语是否只有一个定义（无概念漂移）？
5. 结论是否能从列出的证据与前提推导得出（无跳步）？
6. 是否明确了失效条件/反例（什么情况下结论不成立）？
7. 是否写清“失败算什么”和“失败后做什么”（回滚/补证据/改结论）？
8. 读者在完全不了解上下文的情况下，是否能按文档一次性执行成功？

## 常见陷阱（失败样本）

### 1) 隐含假设（Implicit Assumption）
- **现象**：文档写“运行脚本即可启动服务”，读者照做后报错。
- **根因**：作者默认读者已安装特定版本运行时或已配置环境变量，但未显式声明。
- **复现**：在干净环境执行同样步骤，必然失败。
- **修复**：补一段“前置条件/环境要求/权限要求”，并把关键变量写成可检查项。
- **回归验证**：在无状态环境（例如干净容器）复跑，确认一次性成功。

### 2) 模糊量词（Vague Quantifiers）
- **现象**：需求写“在高负载下保持适当响应速度”，交付后争议不断。
- **根因**：没有阈值与窗口，开发与测试/业务理解不一致。
- **复现**：不同人用不同标准评判“适当”，出现互相否认。
- **修复**：改写为“在 <并发> 下，P99 < <阈值>；错误率 < <阈值>”。
- **回归验证**：将阈值写入测试或验收表，复跑能自动裁决 Pass/Fail。

### 3) 概念漂移（Concept Drift）
- **现象**：上文“用户”指注册用户，下文统计时“用户”变成独立访客（UV）。
- **根因**：同一术语在不同段落含义变化，导致结论不可审计。
- **复现**：计算转化率时分母忽大忽小，甚至出现 >100% 的荒谬结果。
- **修复**：建立术语表，区分并统一用法（例如 RegisteredUser vs Visitor），全文使用同一口径。
- **回归验证**：全局检索关键术语，检查定义与用法是否一致。

## 习惯培养：从“写给自己”到“写给审计者”
- 预设未来会有第三方审查：主动留下可溯源证据与计算过程。
- 维护术语表：为关键概念写出定义、正反例与边界条件，减少术语漂移。
- 留下版本演进：每次修订记录日期、原因与影响面，形成可回溯链路。
