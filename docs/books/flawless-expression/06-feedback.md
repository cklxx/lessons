# 反馈、校对与迭代：把错误变成资产

再精确的初稿也难免遗漏。通过系统化的反馈与校对，可以让表达在每次迭代中更接近“绝对正确”。

## 你将收获什么
- 一个把“收到反馈”变成“可复用改进”的闭环：反馈→修订→回归。
- 一张可复制的“错误资产表”模板：把错误类型、触发条件、修复规则沉淀下来。
- 一份最低门槛校对清单：交付前快速扫掉事实/逻辑/语用三类硬伤。
- 三类失败样本：只收好评不收失败、只改文案不改规则、只修一次不入回归（含修复与验证）。

## 反馈来源与节奏
反馈可以分层获取。写完立刻自检，再隔一段时间复检，避免熟悉性盲区。同侪审阅最好与目标读者角色相似，看他们能否按文档执行任务。涉及法律、安全、财务等高风险领域时，必须让对应专家审计关键段落。完成执行后记录偏差与返工成本，再反推表达中的缺口，这些数据会成为下一版的改进依据。

## 关键工作流（可直接套用）：反馈→修订→回归
1. **收集**：把反馈按“事实/逻辑/语用/格式”分类，并记录触发条件（读者是谁、在哪一步卡住）。
2. **裁决**：每条反馈都要转化为“可验证改动”：要么给出替换句/补充段落，要么明确拒绝（并记录理由）。
3. **回归**：把高频错误沉淀为检查项/模板/规则，下次交付前必须复跑（否则同类错误必复发）。

## 校对清单（可直接复用）
校对时可以沿着七个问题过一遍：每个数字是否标明时间、单位与来源；关键术语是否在前文或术语表中定义；结论是否与证据一一对应，推理是否写出；适用范围、失效条件与风险是否明确；执行者、截止时间与输出格式是否清楚；是否提供附件、链接或版本号以便审计；如果文档被误解，最可能出现的前三种错误是什么，是否已通过措辞或示例避免。

## 示例（可复制）：错误资产表
> 用法：把错误从“一次性修补”升级为“长期免疫”。建议每次重大沟通事故至少入表 1 条。

```markdown
| error_id | 错误类型 | 触发条件（读者/场景） | 现象（怎么错） | 根因 | 修复规则（可复用） | 回归检查（怎么防复发） |
|---:|---|---|---|---|---|---|
| 1 | 事实/逻辑/语用/格式 |  |  |  |  |  |
```

## 迭代与版本策略
把改动留痕能让审阅更高效。使用语义化版本号（如 0.1 草稿、0.9 候选、1.0 发布），在变更记录中写明修改点与审阅人。配合差异视图，审阅者可以聚焦改动而非重读全文。执行后再把实际结果与假设回填，形成“假设→行动→结果→教训”的闭环，高质量的段落与检查表可以沉淀成模板，持续降低未来的撰写成本。

## 与 AI 协作的注意事项
与模型协作时同样需要结构化提示，明确角色、目标、受众、输出格式和约束，让模型减少自由发挥。要求列出推理链与置信度，并手动校验高风险部分；也可以让模型生成反例、误解清单或“最可能的错误执行方式”，再据此改写原文。这样才能让 AI 成为提高正确性的助力，而不是新的风险源。

更稳的做法是把 AI 当作“写作 linter”：只让它做缺失项扫描、歧义定位、反例生成、模板补齐，并要求输出可直接粘贴的替换块；对高风险内容（数字/法律/安全/合规）保持“无证据不采纳”。

## 常见陷阱（失败样本）

### 1) 只收好评，不收失败
- **现象**：大家都说写得好，但上线/执行时仍频繁出错。
- **根因**：反馈来源偏向同温层，缺少“能否执行”的硬反馈。
- **复现**：让目标读者按文档执行一次，立刻暴露缺口。
- **修复**：优先收“失败反馈”（卡在哪、错在哪、返工成本），并按闭环工作流入回归。
- **回归验证**：同类任务二次执行时，卡点数量与返工时长显著下降。

### 2) 只改措辞，不改规则
- **现象**：同类问题反复出现，只是换了说法。
- **根因**：改动没有沉淀为模板/检查项，下一次没人记得复用。
- **复现**：时间一久又写回原来的模糊句式或漏掉关键字段。
- **修复**：把“修复点”写成可复用规则（模板字段、检查清单条目、禁用表达）。
- **回归验证**：发布前检查清单能自动拦下同类错误。

### 3) 只修一次，不入回归
- **现象**：某次事故修完后没有留下痕迹，下一次在另一个文档里复发。
- **根因**：没有错误资产表与回归机制，修复无法被组织记忆。
- **复现**：换一个作者/换一个场景，错误再次出现。
- **修复**：把事故样本入表，补“触发条件→修复规则→回归检查”。
- **回归验证**：后续版本的差异评审能看到该规则被复用并持续生效。
