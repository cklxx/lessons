# 语言与措辞：减少歧义的微调

即使事实和推理正确，模糊语言仍会让表达失真。本章从定义、量化、约束和示例四个角度，说明如何把想法写成可执行的句子。

## 你将收获什么
- 一套把“模糊表达”改写成“可执行句子”的三步工作流。
- 一张可复制的“歧义扫描与替换表”模板（把常见模糊词变成阈值/窗口/责任人）。
- 一份最低门槛检查清单：提交前快速扫掉最容易被误解的句子。
- 三类失败样本：模糊量词、条件缺失、责任不明（含复现与修复）。

## 可操作的语言特征
精准表达通常具备四个特征：术语有定义且有反例，避免概念漂移；描述被参数化，用具体数字、阈值或时间窗口替代“尽快”“大部分”；条件写清触发与排除，如“当 CPU 使用率连续 5 分钟超过 80% 且 QPS>500 时触发扩容”；执行者和责任人明确，例如“由安全团队在 T+1 日 10:00 前审计日志”。这些要素组成了一个可被执行、复现和审计的句子。

## 关键工作流（可直接套用）：歧义扫描→替换→验收
1. **扫描**：圈出模糊词（尽快/适当/多数/显著/稳定/优化/尽量）与隐含条件（环境/权限/范围）。
2. **替换**：把它们改写成“阈值 + 窗口 + 触发条件 + 责任人 + 输出格式”。
3. **验收**：让一个不在上下文里的同事只看这段话就能执行，并能判断 Pass/Fail；否则继续拆分与补条件。

## 典型模糊表达及替代
模糊词可以逐一替换：将“及时反馈”改为“在 24 小时内以邮件形式回复并附决策理由”，把“提升稳定性”落地为“把 99.9% 可用性提升到 99.95%，允许的月度宕机时长不超过 22 分钟”。要求“尽量精简”时直接写“文档控制在 800 字以内，最多两级标题，配 1 张核心示意图”；“优化算法”则改成“在相同数据集上将 F1 从 0.78 提升至 0.82，训练成本不超过当前预算的两倍”。一旦换成可测量的描述，执行与复盘都会容易得多。

## 示例（可复制）：歧义扫描与替换表
> 用法：把关键句子逐条过表。表填不出来，说明你还没把话说清。

```markdown
| 原句（含模糊词） | 读者可能的误解 | 改写后（阈值/窗口/条件/责任/输出） | 验收方式 | 失败判定 |
|---|---|---|---|---|
|  |  |  |  |  |
```

## 句式微调技巧
把结论前置，再补充限定条件，读者可以先理解核心观点。复杂的长句拆成短句或编号步骤，减少语法歧义；对比信息使用对称结构（如“方案 A 更快，方案 B 更省”），差异一眼可见；需要排除项时直接写“本方案不覆盖…”，避免隐含假设。句式上的这些微调，往往比增加新信息更能提升正确性。

## 示例：可执行的 Bug 报告
示例沿着“环境—步骤—期望—实际—影响—优先级”展开，用自然段串联信息。标题写明“Android 15 在夜间模式下切换语言导致应用重启”。环境包括设备、系统与应用版本以及夜间模式的状态。复现步骤按顺序描述：“打开设置页→语言设置，将系统语言从英语切换为法语，再返回应用点击底部导航‘主页’”。期望结果是保持在当前页面并即时切换语言；实际结果则是应用重启并停留在启动页，日志显示资源加载异常。影响面指出夜间模式用户约占 18% 日活，优先级理由写明会话中断会影响支付漏斗，建议定为 P0。整段文字即是一张可以直接执行的缺陷报告。

## 检查清单（最低门槛）
1. 是否存在不可测的模糊词（尽快/适当/多数/显著/稳定/优化/尽量）？若必须使用，是否给出阈值与窗口？
2. 是否写清触发条件与排除条件（什么时候做、什么时候不做）？
3. 是否写清责任人/执行者（谁来做）与截止时间（何时完成）？
4. 是否写清输出格式（交付物是什么：表格/报告/接口/日志）？
5. 是否写清适用范围（平台/版本/人群/权限）与失效条件？
6. 是否能被第三方按文档一次性执行，并判断 Pass/Fail？

## 常见陷阱（失败样本）

### 1) “感觉词”代替门槛
- **现象**：评审会上大家都说“看起来更好”，但没人能说明更好在哪。
- **根因**：缺少可裁决指标与阈值，把主观感受当成验收。
- **复现**：换一批评审者或换一周时间，结论反复横跳。
- **修复**：写成主指标 + 守门指标 + 阈值 + 对比口径，并明确回滚条件。
- **回归验证**：同口径复跑对比表，能自动给出 Pass/Fail。

### 2) 条件缺失导致执行偏差
- **现象**：同一句话不同团队执行出不同结果，互相都认为自己没错。
- **根因**：缺少触发/排除条件与范围边界。
- **复现**：把文档给一个不了解背景的人，让他复述执行步骤，几乎必然补错前提。
- **修复**：补齐条件句与范围说明（平台/版本/权限/数据口径）。
- **回归验证**：用检查表复核关键句子，确保每条都能落到可执行动作。

### 3) 责任不明导致“大家都以为别人会做”
- **现象**：问题长期悬空，最后发现没有人真的接手。
- **根因**：文档只写“需要/应当/建议”，没写“谁在何时交付什么”。
- **复现**：把任务拆到工单/排期时发现缺负责人、缺截止、缺验收。
- **修复**：把动词改为指令式描述：责任人 + 截止时间 + 交付物 + 验收方式。
- **回归验证**：每条行动项都能在任务系统中落地为可分配任务，并可验收关闭。
