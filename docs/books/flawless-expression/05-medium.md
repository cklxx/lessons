# 媒介与格式：为场景选最优载体

表达是否“正确”，不仅取决于内容，还取决于媒介。选择不当会让对方误读或遗漏关键信息。本章用决策树思路说明如何选媒介，并给出场景化的写作模板。

## 你将收获什么
- 一张“媒介选择决策树”（纯文本版）：用四个问题快速选载体。
- 三个可复制模板：决策摘要、执行任务单、审计记录（版本/证据/回滚）。
- 一份最低门槛检查清单：避免信息密度不匹配导致误读与漏读。
- 三类失败样本：用会议替代文档、用长文替代工单、用截图替代证据（含修复与回归）。

## 选择媒介的决策树
判断媒介前先问四个问题：目标是决策、执行、对齐还是记录；是否需要实时互动还是可以延迟处理；内容是否依赖图表、数据表或代码；是否需要版本历史、批注和可搜索性。答案决定载体：需要快速澄清与共同建模时用会议、电话或视频，并准备议程与纪要；要审计轨迹或跨时区协作时，邮件、PRD、决策备忘录或设计说明更合适；信息需要标准字段和统计时使用工单或 Issue 这类结构化表单；要让机器直接执行时，用脚本、配置或 API 契约最可靠。

### 决策树（纯文本可复制）
```text
1) 目标是什么？
   - 决策：优先“决策摘要/备忘录”，先结论后证据与风险。
   - 执行：优先“工单/任务单”，字段齐全，验收可裁决。
   - 对齐：优先“对齐文档 + 会议纪要”，把分歧与决策记录下来。
   - 记录/审计：优先“版本化文档/日志”，能追溯到证据与变更原因。

2) 需要实时互动吗？
   - 需要：开会，但必须产出纪要与行动项（否则会议等于没发生）。
   - 不需要：异步文档优先，降低信息丢失与重复沟通成本。

3) 内容依赖结构化字段吗？
   - 依赖：用表单/工单/表格；避免在长文里埋字段。
   - 不依赖：用邮件/备忘录/文档。

4) 是否需要可搜索与版本历史？
   - 需要：文档/仓库/工单系统；不要只发截图或口头承诺。
```

## 关键格式要素
无论选择哪种媒介，都要在显眼位置写出日期、版本、作者、审核人、适用范围与失效条件。正文使用清晰的标题与编号，让阅读路径可预测，并用必要的示意图、时序图或流程图辅助理解，同时在图旁附上文字说明与图例。源文件、实验数据或日志应放在附件或仓库中，正文用相对路径引用，方便审计和复现。

## 场景化模板
一页纸决策摘要可以用“结论—背景—方案与优缺点—决策理由—风险与复盘时间”组成，控制在一页内即可。技术设计评审包常从问题与目标指标开始，补充当前架构与瓶颈的图示，再列出方案 A/B/C 的风险、成本与性能对比，最后写出观测与回滚计划。冲刺任务单则围绕“用户故事、验收条件、负责人、截止时间、依赖、风险与测试上线检查项”展开。调研资料包需要把研究问题、范围与排除项写清，并附上证据矩阵摘要（来源、时间、可信度、主结论）、关键图表或数据表的可再现链接与文件名，补充冲突证据、未解问题以及下一步验证计划。写作时用连贯的段落串联这些字段，读者不需要在碎片化列表中拼接信息。

### 示例（可复制）：一页纸决策摘要
```markdown
# <决策标题>
- 日期：YYYY-MM-DD
- 版本：v0.1 / v0.9 / v1.0
- 作者/审核：<owner> / <reviewer>

## 结论（先给答案）
<一句话决策>

## 背景与约束
- 背景：
- 约束：<时间/预算/合规/依赖>

## 证据（最小单元）
- <证据 1（链接/数据/日志）>
- <证据 2>

## 备选方案与取舍
- 方案 A：<优点/缺点/成本/风险>
- 方案 B：<优点/缺点/成本/风险>

## 风险与回滚
- 失败判定：<主指标/守门指标阈值>
- 回滚动作：<开关/降级/撤回>
```

### 示例（可复制）：执行任务单（工单/Issue 版）
```markdown
# <任务标题>
## 背景
## 目标（可量化）
## 范围 / 非范围
## 验收标准（Given/When/Then）
## 失败判定（阻断条件）
## 风险与回滚
## 证据留档（路径/链接）
```

## 检查清单（最低门槛）
1. 载体是否匹配目标（决策/执行/对齐/记录）？
2. 是否需要实时互动？如果开会，是否产出纪要与行动项？
3. 是否包含日期、版本、作者、适用范围、失效条件？
4. 是否把关键字段结构化（避免散落在长段落里）？
5. 是否提供证据链接/路径（而非仅截图或口头承诺）？
6. 是否写清失败判定与回滚动作？

## 常见陷阱（失败样本）

### 1) 用会议替代文档
- **现象**：会上都点头，事后各做各的，复盘时找不到决策依据。
- **根因**：会议没有纪要与行动项，信息未落地成可追溯资产。
- **复现**：一周后换一批参会者，结论被重新争论一遍。
- **修复**：会前有议程，会后 24 小时内发纪要：结论、分歧、行动项、截止、证据与回滚。
- **回归验证**：任何关键决策都能在文档/工单中被检索到，并能追溯到证据与版本。

### 2) 用长文替代工单
- **现象**：需求写得很长，但开发不知道要做哪些具体任务与验收。
- **根因**：缺少结构化字段与可裁决验收，导致执行断层。
- **复现**：不同开发拆出不同任务清单，交付物不可对齐。
- **修复**：把执行部分沉到工单/任务单：范围、验收、失败判定、责任人与截止。
- **回归验证**：任务单能直接分配与关闭，且验收可裁决。

### 3) 用截图替代证据
- **现象**：报告里全是截图，没有原始数据或可复算路径。
- **根因**：把“展示”当成“证明”，缺少可复现的证据链。
- **复现**：换一台机器或换一个时间点，截图无法再生成，结论不可审计。
- **修复**：证据必须包含来源与可复跑路径（链接/文件/口径/脚本/查询条件）。
- **回归验证**：第三方能按文档复现关键数字与结论（趋势一致、阈值判断一致）。
