# 第 6 章：UI 设计：把体验做成可维护资产

在从零到一的产品开发阶段，用户界面（UI）设计往往面临一个核心挑战：如何从最初的探索性原型，逐步演化成一套既美观又易于维护，同时能支撑高速迭代的产品资产？许多团队将UI设计视为美化或实现稿件，而忽略了其作为产品核心体验承载者的深层价值。这种偏见导致的问题显而易见：产品迭代中，UI常常成为瓶颈，每次修改都像推倒重来，用户体验也随之变得碎片化、不连贯，甚至出现严重的用户流失。

本章将深入探讨，在0到1阶段，UI设计如何超越简单的视觉呈现，成为构建可持续产品竞争力的关键一环。我们的目标不仅仅是让界面好用，更是要让它经得起迭代，将不确定性压缩在最小，确保用户能清晰地感知下一步、在遇到问题时能顺利恢复，并且每一次产品改动都能在统一的规则下平稳回归。我们将把UI从截图工程提升为可维护资产，构建一套能复用、能解释、能回归、能适应下一轮迭代的设计系统，从而为后续的工程化与规模化发展奠定坚实基础。

## UI 资产化：从灵感到系统的蜕变

本章的定位介于验证与打磨和工程化之间。在产品核心价值主张（价值闭环）已被初步验证之后，UI的使命不再仅仅是展示功能，而是要把验证过的体验沉淀为一套可维护的资产，为后续的工程实现提供清晰、统一的指引。这意味着，我们不仅要关注画得漂亮，更要聚焦于如何让UI具备高内聚、低耦合的特性，如同高质量的代码一般，易于理解、修改和扩展。

读完本章，你将掌握输出可实现、可验收、可回归UI规格的核心方法。每一次UI改动，你都能清晰地回答以下问题：它改善了哪个用户体验断点？它是否引入了新的失败场景？它是否能在下轮迭代中继续保持可维护性？我们倡导一套文字规格优先的工作流，先明确任务、信息层级、状态与错误恢复，再考虑视觉表现；同时，你将学会一份最小可访问性（a11y）验收清单，将可用提升为发布产品的基本门槛；最终，你将掌握把界面沉淀为资产的方法论，包括组件与变体的设计、状态基线的定义、文案规范的制定以及边界与例外情况的处理。

### UI 设计的本质是降低认知与维护成本

UI设计的深层本质在于通过视觉和交互手段，最大程度地降低用户的认知成本和产品的维护成本。这两者是互为表里、相互促进的：清晰、一致、可预测的UI能让用户毫不费力地完成任务，减少摸索和困惑，从而降低认知负担；而一套有规范、可复用的UI系统，则能极大地提高开发效率，减少重复劳动，降低因随意改动导致的维护成本。

先把这章要解决的核心矛盾和边界说清楚。

作为产品经理或设计师，本章旨在赋能你产出高质量的UI规格，确保团队的每一次UI迭代都是有章可循、有据可依的。你需要能够清晰地阐述每次UI调整背后的用户痛点和商业价值，例如，某个界面的改动是为了优化新手引导流程中特定步骤的转化率，还是为了减少用户在复杂表单填写时的错误率。此外，你还需确保改动不会引入新的体验断点或维护负担。最终目标是让UI设计能够被量化评估，通过任务完成率、平均耗时和错误恢复率等指标来衡量其有效性，并与团队在相同口径下进行沟通和协作。

再把从问题到方案再到验收的推演补齐。

UI从最初的灵感闪现到最终成为可维护的产品资产，并非一蹴而就，而是一个环环相扣的论证链条。这个链条始于对**任务与信息层级**的深刻理解，它决定了用户如何感知和处理信息。在此基础上，我们需要细致考量**状态与错误恢复**机制，确保产品在任何场景下都能提供明确反馈和解决方案。随后，通过抽象与复用，我们构建**组件与变体**，为界面元素赋予结构化的生命。**文案与一致性**则如同产品的声音，统一的语言风格能增强用户信任感。**可访问性门槛**（a11y）则是保证产品普适性的基石，它确保了所有用户，无论其能力如何，都能顺畅使用。最后，**回归基线**为迭代提供了安全网，保证每次更新都在可控范围内。缺少其中任何一环，UI就可能沦为截图工程——表面光鲜，实则功能不稳定、难以维护，无法应对真实世界的复杂挑战。

最后落到可执行的门禁、证据与回滚。

对UI的验收不应止步于视觉对齐的表面功夫，更要深入到功能和体验的本质。一次成功的UI验收，意味着产品在以下几个维度都达到了预期标准：首先，**关键任务是否更短、更清晰、更可预测**？用户能否在最少的步骤内，以最直观的方式完成核心操作。其次，**关键状态是否齐全**？例如，加载中、空数据、操作成功、操作失败、无权限等各种状态，是否都提供了明确的视觉反馈和交互引导。再者，**键盘与辅助技术是否可用**？这是可访问性的最低门槛，确保了产品对特殊用户群体的友好度。最后，**文案与布局在真实约束下是否成立**？我们必须考虑极端情况，如长文案是否会溢出，移动端在弱网环境下的表现如何，确保设计在各种现实场景中都能保持可用性和稳定性。

## 方法论速览：先写规格，再做风格，再沉淀组件

### 1) 先写文字规格（让争论有裁判）

在UI设计的早期阶段，很多团队会直接跳到视觉稿的绘制，但这种做法往往导致后续的反复修改和沟通成本。我们倡导一种文字规格优先的工作方式，即在任何视觉设计之前，先用清晰的文字来定义页面的功能、信息层级和交互逻辑。这并非仅仅是写一篇产品文案，而是建立一套严谨的文档，明确核心用例、用户预期、以及系统在不同状态下的响应。这种方法的核心在于将UI设计问题前置为逻辑问题，让团队的讨论和决策有据可循，避免陷入我觉得好看或我觉得不好看的主观争论泥潭，从而显著提高设计效率和质量。

**案例一：任务驱动的 B2B SaaS 平台**

某新兴的B2B数据分析SaaS平台，在初期面临客户对数据报表定制功能复杂度的抱怨。产品团队在没有明确文字规格的情况下，多次修改UI，结果总是顾此失彼。例如，为了增加筛选功能，导致页面元素过多，用户迷失；为了简化操作，又隐藏了关键配置，让高级用户感到不便。

后来，团队采纳了文字规格优先的方法。他们首先为创建自定义报表这一核心任务撰写了详细的文字规格：

*   **页面目标**：用户在这里通过拖拽、筛选完成自定义数据报表的生成与预览，并导出。
*   **主 CTA**：生成报表/保存模板；失败时提示数据源异常，请检查配置或联系客服。
*   **信息层级**：主信息：报表预览区；次信息：筛选条件设置区、字段选择区；辅助信息：操作历史、帮助文档入口。
*   **状态清单**：
    *   **空状态**：显示尚未创建任何报表，点击此处开始创建，附带示例链接。
    *   **加载中**：报表预览区显示骨架屏，筛选区禁用。
    *   **成功**：预览区显示完整报表，导出按钮可用。
    *   **失败**：预览区显示错误信息报表生成失败：[错误码]，请重试或查看日志，提供重试和联系客服按钮。
    *   **无权限**：显示您无权访问此功能，请联系管理员开通权限。
*   **错误恢复**：报表生成失败时，提供重试、检查数据源配置链接和联系客服入口。
*   **约束**：支持桌面浏览器，数据量可达百万级，报表生成时间最长 30 秒。
*   **验收**：新用户在 2 分钟内成功生成一份基础报表，错误率低于 2%。

通过这份文字规格，设计师和开发工程师在视觉实现前，就对功能的各个环节、可能出现的异常以及如何处理有了统一的认知。这种前置的、逻辑驱动的规格设计，使得后续的视觉设计更加顺畅，大大减少了返工，确保了最终交付的UI既符合用户预期又兼顾了鲁棒性。

**模板：页面文字规格（最小可验收）**

| 模块 | 你要写什么 |
| --- | --- |
| 页面目标 | 一句话：用户来这里要完成什么 |
| 主 CTA | 一句话：点击后发生什么；失败如何提示 |
| 信息层级 | 3 层以内：主信息/次信息/辅助信息 |
| 状态清单 | 空/加载/成功/失败/无权限（每个状态的文案与入口） |
| 错误恢复 | 重试/返回/求助/联系客服/查看示例（至少 1 个） |
| 约束 | 设备/网络/权限/数据规模（影响交互与文案） |
| 验收 | 任务完成率、耗时、错误恢复率（同口径） |

### 2) 再谈风格（风格服务于信息层级）

UI风格并非无关紧要，但它必须服务于核心目的：清晰地传达信息层级，并引导用户完成关键任务。颜色、字号、间距、对比度以及动效等视觉元素，都应被视为强化下一步是什么的工具。例如，一个主操作按钮应通过其颜色、大小和位置，在页面上形成最强的视觉重心，明确告知用户哪个是首要的行动点。次要操作则应保持较低的视觉权重。如果风格喧宾夺主，例如使用了过于花哨的动效导致用户注意力分散，或者对比度不足让关键信息难以辨识，那么这说明风格正在掩盖结构性问题，用美观牺牲了可用性。在0到1阶段，保持简洁和功能性至关重要，过度追求设计感而牺牲清晰度是本末倒置。

### 3) 把组件当成长期资产

在产品快速迭代的0到1阶段，虽然尚不需要一套完整的企业级设计系统，但建立一个不会失控的最小集合的组件库是至关重要的。这就像软件开发中的模块化思想，将UI元素抽象为可复用的组件，从而提高开发效率、确保设计一致性并降低维护成本。组件不仅是视觉元素，更是行为模式的载体，它包含了不同状态下的表现和交互逻辑。

这个最小集合通常包括基础组件（如按钮、输入框、对话框、列表、提示消息等）和一些业务场景特有的组件（如空状态提示、加载骨架屏等）。关键在于为每个组件明确定义其变体（如大小、颜色、形态）和所有可能的状态（如默认、禁用、悬停、聚焦、加载、错误、成功等），并规定其在不同状态下的行为和视觉反馈。业务页面在构建时，应严格遵循组合组件，不复制样式的原则，尽量通过组合现有组件来满足需求，而非每次都从头创建新的视觉样式。这种规则优先的方法，能有效防止UI随着迭代而逐渐碎片化、风格不一。

**案例二：从截图UI到组件驱动的转型**

某初创公司在初期开发时，为了追求速度，每个页面都由设计师独立完成，前端工程师也只是截图还原。随着功能日益增多，问题浮出水面：同一功能，不同页面有不同的按钮样式；一个设计改动，需要在十几个页面手动修改；更糟糕的是，新入职的设计师和工程师无所适从，导致开发效率骤降，Bug层出不穷。团队称之为截图UI灾难。

痛定思痛后，他们决定转型，建立了一个轻量级的组件库。第一步是梳理出现有UI中出现频率最高的基础元素，如按钮、输入框、对话框、表格等，并将其抽象为组件。然后，他们为每个组件制定了详细的规格：

*   **组件：Button**
    *   **变体**：Primary（主操作，蓝色）、Secondary（次要操作，灰色）、Danger（危险操作，红色）、Link（链接按钮）。
    *   **状态**：默认、悬停、按下、禁用、加载中（禁用并显示Loading图标）。
    *   **交互与反馈**：
        *   键盘可触达（Tab键可切换焦点，Enter键可触发）。
        *   加载中状态时，按钮文案变为处理中...，并显示加载动画，按钮不可点击。
    *   **禁止项**：
        *   在同一页面内使用超过两种不同主色调的按钮。
        *   在未经组件库批准的情况下，自定义一套新的按钮样式。
*   **组件：Input**
    *   **变体**：Text、Password、Number、Search。
    *   **状态**：默认、聚焦、错误（红色边框+错误提示文案）、只读、禁用。
    *   **交互与反馈**：
        *   聚焦时显示蓝色边框。
        *   输入错误时，在输入框下方显示清晰的错误提示文案。
    *   **禁止项**：
        *   仅使用占位符（placeholder）作为输入框的唯一标签提示，必须有独立的`label`标签。
        *   错误提示文案模糊不清或无法引导用户恢复。

通过这样的组件规格定义，团队成员之间有了共同的设计语言和实现规范。新的UI需求不再是画一个新页面，而是组合现有的组件来满足需求。这不仅显著提升了开发效率，也确保了产品整体视觉风格和交互体验的一致性，有效避免了截图UI的再次出现，将UI真正转化为可维护的长期资产。

**模板：组件规格（变体与状态）**

| 组件 | 变体 | 状态（必须覆盖） | 交互与反馈 | 禁止项 |
| --- | --- | --- | --- | --- |
| Button | primary/secondary/danger | 默认/禁用/加载 | 键盘可触达；加载时不可重复提交 | 在页面内自定义一套新按钮 |
| Input | text/password | 默认/聚焦/错误/只读 | 错误提示可理解且可恢复 | 仅用占位符代替标签 |

**读者练习：**

1.  **分析现有产品：** 选取你正在开发或熟悉的任意一个产品（可以是内部工具或公开应用），识别出其中3个你认为最关键的页面或功能模块。尝试为其中一个页面撰写一份页面文字规格，包含页面目标、主CTA、信息层级、状态清单和错误恢复。
2.  **组件拆解与规范初稿：** 从你的产品中选择一个常用的UI元素（如按钮或输入框），尝试为其定义一份组件规格，包括所有你观察到的变体、状态，以及对应的交互反馈和你认为应该禁止的设计行为。
3.  **识别可访问性断点：** 使用键盘（仅限Tab、Enter、Esc键）浏览你选定的产品，尝试完成一个核心任务。记录下你在哪些环节感到困难或无法操作，思考这与最小a11y验收清单中的哪些项相关。
4.  **长文案挑战：** 设想你产品中的某个关键提示或表单字段，其文案长度增加了50%或100%（例如，中文文案翻译成德文）。思考并记录这种变化会如何影响当前页面的布局和美观，是否会导致信息溢出或遮挡核心操作。
5.  **设计评审准备：** 准备在下次设计评审会上，向团队成员解释你所定义的页面文字规格和组件规格如何帮助团队提高效率和一致性，并预设可能遇到的反对意见和应对策略。

### 可访问性（a11y）：把可用变成发布门槛

可访问性（Accessibility，简称a11y）常常被误解为为少数特殊人群提供额外功能，但在产品实践中，它远不止于此。将a11y纳入设计和开发流程，实际上是提升产品整体健壮性、普适性和用户体验的关键。一个无障碍的产品，不仅能服务于残障用户，也能让键盘用户、弱网环境下的用户、使用老旧设备的用户，甚至临时性身体不便（如手臂受伤）的用户都能顺畅使用。它迫使我们在设计时更深入地思考交互的底层逻辑，将用户路径做得更加扎实和严谨。把a11y从锦上添花变为发布门槛，意味着从产品立项之初就将其视为核心质量指标，而不是在产品上线后才修补的补丁。

**最小 a11y 验收清单（建议作为门槛）**

在0到1阶段，我们无法追求全面完美的a11y，但可以设定一个最小的、可操作的验收清单，作为产品上线的硬性门槛：

*   **键盘可完成关键任务**：所有核心操作，如导航、表单填写、提交、弹窗关闭等，都必须能够仅通过键盘（Tab键切换焦点，Enter键确认，Esc键取消）顺畅完成。这要求我们审视Tab键的焦点顺序是否符合逻辑，是否存在键盘黑洞让用户卡住。
*   **焦点可见且顺序合理**：当用户使用键盘Tab键切换焦点时，当前被聚焦的元素必须有清晰的视觉指示（如蓝色边框），且焦点移动顺序应与页面内容阅读顺序保持一致。避免焦点在页面上跳跃或迷失，导致用户困惑。
*   **文本对比度达标**：核心信息（如标题、正文）和关键交互元素（如按钮文案）的颜色与背景色之间必须有足够的对比度，以确保视力障碍用户和在光线不佳环境下使用的用户能够清晰辨识。这通常遵循WCAG 2.2 AA级标准。
*   **表单有明确标签与错误提示**：所有输入框都必须拥有明确的可见标签（而非仅仅是占位符），并且当输入发生错误时，错误提示信息应清晰、易懂，且能指导用户如何纠正错误（例如，密码长度至少8位，包含大小写字母和数字）。
*   **动效可克制**：产品的动画和动效不应成为传递关键信息的唯一手段，且用户应有能力选择减少或关闭不必要的动效。避免使用可能引起眩晕或不适的闪烁、快速切换等动效。

**失败案例：某新电商平台的无障碍陷阱**

一家雄心勃勃的初创电商平台，在早期产品发布时，为了追求视觉潮流和酷炫动效，忽略了a11y规范。他们的产品中大量使用了仅依赖鼠标悬停才能触发的菜单、对比度极低的灰色文字，以及缺乏明确焦点指示的自定义表单控件。

上线后不久，用户反馈平台存在诸多使用障碍：

*   **老年用户流失**：许多老年用户反映，由于视力下降，页面上的文字难以阅读，导航菜单怎么都点不开，导致他们无法完成购物。
*   **键盘用户无法操作**：一位长期使用键盘操作电脑的残障用户在社交媒体上发文，指出该平台的购物车结算页面，由于Tab键焦点顺序混乱且提交按钮没有键盘触发事件，导致他根本无法完成支付，最终只能放弃购买。
*   **品牌声誉受损**：这些负面反馈迅速在社区传播，平台被指责缺乏社会责任，导致品牌形象受损，甚至有用户考虑提起投诉。

面对压力，公司不得不投入大量资源进行紧急整改，重新设计了大量UI元素，并强制要求所有新功能必须通过a11y检查。这次经历让他们付出了巨大的代价，不仅错失了早期市场机会，还耗费了数百万的开发成本来修复本应在设计初期就避免的问题。这个案例深刻说明，将a11y视为后期补丁而非早期门槛，最终的成本将远超预期。

### 多语言与多端：把真实约束提前

在0到1阶段，产品往往在理想化的环境下被设计和测试：标准的显示器、充裕的网络带宽、以及简洁明了的中文文案。然而，一旦产品走向真实世界，它必须面对各种严苛的约束：不同尺寸的移动设备、不稳定的网络环境、以及长度差异巨大的多语言文本。忽视这些真实约束，是导致产品在上线后频繁爆雷的常见原因。将多语言（i18n）和多端适配（响应式）的考量前置，不仅能避免后期巨大的返工成本，更能确保产品在各种环境下都能提供稳定、可靠的用户体验。

**最小检查清单（i18n/多端）**

为了应对多语言和多端带来的挑战，以下是一份在设计初期就应纳入考量的最小检查清单：

*   **长文案不溢出、不遮挡 CTA**：假设你的产品需要支持德语、俄语等语言，这些语言的词汇长度通常远超中文和英文。在设计时，必须预留足够的空间来容纳超长文案，并确保长文案不会导致布局混乱、文本溢出容器，更不能遮挡住任何关键的召唤行动（CTA）按钮。理想情况下，至少需要准备一组超长文案的样例，并在不同屏幕尺寸下进行视觉验证。
*   **移动端关键任务不需要精确点击**：在移动设备上，用户的操作通常是通过手指触摸完成的，相比鼠标，手指的精确度要低得多。因此，所有关键操作按钮、链接和可交互区域，都必须有足够大的点击热区（通常建议最小尺寸为44x44像素），并确保它们之间有足够的间距，避免用户误触。尤其要注意，在移动端避免小按钮挤在一起，或关键信息在小屏幕上过于密集难以分辨。
*   **弱网/慢设备下加载态与重试入口明确**：在网络状况不佳或设备性能较低的环境下，数据加载缓慢是常态。此时，产品必须提供清晰的加载指示（如骨架屏、加载动画），而不是让用户面对一个空白或冻结的界面。更重要的是，当加载失败或操作超时时，必须提供明确的错误提示和重试入口，告知用户问题所在并引导他们如何恢复，避免用户因不确定性而直接放弃使用。

### AI 怎么用在 UI 上（但不替你做判断）

人工智能在UI设计流程中，扮演的更多是高效的辅助者角色，而非替代人类设计师进行决策。AI擅长处理重复性、规则性的任务，生成多样化的备选方案，并进行一致性检查。但对用户心理的洞察、复杂场景的权衡以及最终的创意判断，依然离不开人类的智慧。

在实际应用中，我们可以将AI的力量集中在以下几个方面：

*   **生成备选与评估断点**：让AI根据用户需求和潜在痛点，生成多个版本的页面文案、引导提示或操作流程。在生成时，要求AI同时说明每个版本对应哪类失败断点，例如，某个文案版本是为了解决用户不理解下一步操作的问题，另一个版本是为了解决用户在错误状态下无从下手的问题。这能帮助设计师快速获得多种视角，而非从零开始构思。
*   **一致性巡检**：利用AI进行自动化的一致性检查。例如，让AI审查产品中的所有术语，确保其在不同页面和模块中保持统一；检查所有按钮的动词是否一致（例如，是提交还是确认，是删除还是移除）；以及扫描所有错误提示信息，评估它们是否都具备可理解性和可恢复性，能否明确告诉用户问题出在哪里以及如何解决。
*   **a11y 风险预警与建议**：AI可以根据预设的a11y清单（如WCAG标准），自动识别UI中的潜在风险点，例如对比度不足的文本、缺少alt文本的图片、键盘焦点跳跃等。更进一步，我们可以要求AI针对发现的每一个风险点，输出一句可直接改写的、具体的改进建议，而不是泛泛的理论指导。例如，AI可能会建议：将按钮‘确定’的背景色从#F0F0F0调整为#4A90E2，以满足WCAG AA级对比度要求。

#### 避免 AI 高保真原型误导

在使用人工智能进行原型设计与界面草稿时，高保真原型易给人造成项目进展顺利的错觉，从而误导团队对真实进度的判断。为规避此风险，验收时必须强制覆盖完整的任务流、各种异常流及所有可能的状态。测试时应聚焦于真实的复杂设计场景，而非仅限于简单页面或单一功能。通过模拟真实的用户交互路径和潜在问题，才能全面评估设计的健壮性与可用性，避免过早乐观。[82][83]

#### 对话式界面中的提示词控件化

对话式界面可通过将提示词转化为可交互的控件，使用户无需掌握复杂的提示词工程知识，也能精准表达其需求和约束。例如，对于图片生成，可提供下拉菜单选择风格，滑块调整细节程度，颜色选择器指定主色调等。这些控件将潜在的提示词要素可视化并标准化，极大地降低了用户门槛。整个交互流程应强调控制权、支持可撤销操作、允许回溯历史状态，并提供清晰的反馈闭环，确保用户始终了解其指令对结果的影响。[80][81]

### 复现检查清单（本章最低门槛）

要确保本章所学落地生根，请以以下清单作为产品发布的最低门槛：

*   **核心页面规格完备**：确保每个关键页面都拥有清晰的页面文字规格文档，其中必须包含任务目标、状态清单和错误恢复入口。缺少其中任何一项，都应视为该页面未完成设计。
*   **任务走查通过**：进行至少一轮的无指导用户任务走查。这意味着，让一个不熟悉产品的用户在没有任何外部指导的情况下，独立完成产品的核心任务。用户必须能够自如地完成任务，并且在遇到任何失败或障碍时，能够依靠产品提供的指引进行自救，顺利恢复到正常路径。
*   **a11y 最小门槛满足**：产品必须满足至少的a11y基本要求：所有关键任务都可通过键盘操作完成，键盘焦点可见且顺序合理，所有错误提示都清晰、可理解并能引导用户恢复。

### 常见陷阱（失败样本）

产品设计和开发中，有几类与UI相关的坑是团队反复容易踩到的：

1.  **现象**：界面看起来非常精美、富有艺术感，但用户在实际使用时却找不到关键的操作入口，或者不知道下一步该做什么。
    **根因**：设计师在追求视觉效果时，过度关注美观而忽视了功能性和信息层级。视觉装饰元素压倒了信息优先级，导致核心功能被淹没，用户认知成本极高。产品缺乏对下一步是什么的明确视觉引导。
    **修复建议**：重新回到文字规格优先的原则。首先梳理页面的核心任务和信息层级，明确主次关系。通过调整布局、大小、颜色等视觉元素，强化主操作的视觉权重。在确保信息层级清晰、用户路径明确的前提下，再考虑优化视觉风格。

2.  **现象**：随着产品的不断迭代，UI变得越来越混乱，不同页面出现样式不统一的按钮、输入框，或同一功能有多种不同的实现方式。每次修改一个UI元素，都需要在多个地方手动调整，导致开发效率低下，bug频发。
    **根因**：团队没有建立一个最小组件集合的单一来源。设计师随意创建新的样式，开发工程师也随手复制粘贴代码，导致UI组件逐渐碎片化，失去了统一性。
    **修复建议**：立即启动组件库的建设。识别产品中所有可复用的基础UI元素，并为它们定义详细的组件规格，包括变体、状态、交互规则和禁止项。强制团队遵循规则优先的原则，任何新的UI需求都必须通过组合现有组件来满足，或在组件库中进行规范化扩展。

3.  **现象**：产品在内部测试时运行良好，但在上线后，用户反馈出现大量的加载慢、页面崩溃、操作失败后无从恢复等问题，导致用户流失和差评。
    **根因**：产品设计和开发团队往往只在成功路径下进行设计和测试，将失败场景（如网络错误、数据异常、权限不足）视为工程实现的细节，而不是用户体验的重要组成部分。导致产品在真实世界的复杂环境中缺乏鲁棒性。
    **修复建议**：将失败与恢复视为用户体验的核心环节。在页面文字规格和组件规格中，明确定义所有关键状态（空、加载、失败、无权限）下的UI表现和错误恢复入口。将这些失败场景的测试纳入到验收标准中，确保产品在任何异常情况下都能提供明确的反馈和可行的解决方案。

## 交付物清单与验收标准

本章结束时，你应该能输出以下交付物，并作为团队内部的设计验收标准：

*   **页面文字规格**：至少为产品中的3个关键页面提供完整的文字规格文档，明确其目标、CTA、信息层级、状态清单和错误恢复策略。
*   **组件规格**：为产品中至少5个基础UI组件（如按钮、输入框、提示框）定义详细的规格，包括其变体、状态、交互反馈和使用规范。
*   **a11y 与多端检查记录**：提供一份覆盖关键任务的a11y和多端检查记录，记录发现的问题点和对应的改进方案。
