## P0（下一轮优先，影响全书一致性）
1. **把示例（可复制）升级为标准块**：每章至少 1 个端到端示例，按 `A-templates.md` 的 A.5 模板补齐字段；验收：示例块包含（前置条件/步骤/验证命令/预期输出/失败判定/回滚），且读者能直接照抄结构替换占位符。
2. **每章至少 1 个关键流程图（先不依赖 Mermaid）**：用纯文本流程图（` ```text `）/表格/状态机表呈现该章关键闭环；验收：图/表必须覆盖输入、关键分支、输出与至少 1 个失败回退点。
3. **常见陷阱模板化**：每章至少 3 个失败模式，结构固定为现象 → 根因 → 复现 → 修复 → 回归验证；验收：每条至少给出一个可操作的验证方式（命令/日志片段/指标口径之一）与看到什么算修好。
4. **术语表扩充 + 禁用同义词策略**：补齐 `glossary.md`（建议 ≥ 25 个核心词条），并增加本书禁用/优先用法小节；验收：全书不再出现同一概念多译混用，且首次出现处有链接。
5. **全书前置假设与环境约定**：在 `index.md` 或 `preface.md` 明确：基准环境/工具链版本、`make ...` 为读者项目仓库中的约定示例、哪些内容是伪代码；验收：读者开读前就能理解本仓库不提供可执行工程代码，但提供可复用的落地范式与自检步骤。
6. **命令命名风格对齐（文档示例）**：统一正文示例中的任务命名（如 `make <domain>-check` / `make <domain>-eval`），并在 `C-checklists.md` 给出主推命名 + 兼容别名策略；验收：同类检查跨章不再出现多套命名。
7. **补齐资料笔记覆盖面**：将 `filtered/top-resources.md` 中的关键条目逐步整理进 `materials/.../notes/`（一条资料一页）；验收：至少覆盖每章 1–2 份权威资料，并在章节末尾资料笔记（可选）里能回链。
8. **统一合规/成本提示口径**：对涉及爬取、数据、日志、工具调用、生产部署的章节补齐合规与成本提醒（PII、平台条款、日志脱敏、调用预算）；验收：读者不会因为照做而踩 ToS/隐私红线，且能预估验证/评测成本。

## P1（提升可读性与可迁移性）
1. **关键流程补结构图**：每章至少 1 张（Mermaid 架构/时序/状态机三选一；若不启用 Mermaid，则用等价的纯文本流程图/表格替代）；验收：图表在 Markdown 中可阅读且不依赖外部截图文件。
2. **失败样本驱动写作模板化**：把常见陷阱改写为现象 → 根因 → 复现 → 修复 → 回归验证；验收：每章至少覆盖 3 个可复现失败模式。
3. **把 Prompt Library 抽到附录并回链**：将正文里的高频 Prompt 汇总到 `B-prompts.md`，正文只保留 1 个代表性示例；验收：正文示例与附录模板互相链接，读者能快速复用。
4. **把验收标准写成可检查项**：对交付物清单与验收标准增加最低门槛/可选增强；验收：每条验收项都能对应一个验证动作（命令/日志/报告/截图）。
5. **章节间交叉引用**：为关键概念加详见第 X 章；验收：每章至少 2 处跨章链接，且不产生循环依赖式跳读。

## P2（长期优化）
1. **一致的代码片段风格**：统一语言标注、注释密度、占位符写法（`path/to/file`、`YOUR_TOKEN`）；验收：随机抽 3 章对比，代码块风格一致。
2. **增加安全与合规边界对照表**：汇总 PII、凭据、支付与模型数据边界的处理建议；验收：表格能直接作为项目安全 checklist 使用。
3. **补读者路径与角色视角导读**：在总览页补 3 条阅读路径（产品/工程/模型）；验收：读者能按目标（上线/提质/降本）快速定位章节。
