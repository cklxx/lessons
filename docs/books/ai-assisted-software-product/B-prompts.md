# 附录 B：提示词与工作流配方

> 目标不是写出更厉害的提示词，而是把提示词变成可复用的工作流：**输入规范 → 输出格式 → 评审清单 → 回归机制**。

## B.1 通用提示词骨架（推荐）

```text
角色：你是谁（资深 PM/架构师/设计师/SRE…）
背景：产品是什么、当前阶段、已有资料链接或摘要
目标：本次要达成什么（可量化、可验收）
约束：不能做什么、必须满足什么（技术/合规/成本/时间）
样例：给 1–3 个正反例（越真实越好）
输出：指定格式（表格/JSON/列表），以及不确定点/需要追问的问题
```

## B.2 需求 → PRD 工作流

1. 输入：访谈摘录（原话）+ 目标用户 + 业务目标
2. 输出：主题聚类 + 假设 + 实验计划
3. 评审：证据是否充分？是否有替代路径？指标是否可量化？
4. 回归：把失败样例加入需求回归集

## B.3 原型 → UI 规格工作流

1. 输入：页面树 + 关键任务 + 组件清单
2. 输出：状态清单（空/加载/错误/无权限）+ 接口问题列表
3. 评审：是否覆盖异常流？是否可实现？是否可测试？

## B.4 代码生成的约束式提示

- 先输出接口/类型定义，再输出实现
- 必须包含测试用例
- 必须说明边界条件与错误处理
- 禁止引入新增依赖（除非明确允许）
