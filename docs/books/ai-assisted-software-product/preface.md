# 序言：一人即是一支队伍

凌晨两点，屏幕荧光闪烁。你有一个可能改变某个细分领域的想法，但疲惫先于兴奋抵达：产品、设计、前后端、部署、运维……过去，这些意味着一支团队；而你只有一个人。今天，算式变了：AI 把“实现”压缩到更低的边际成本，但也把“走错路”推到更高的速度。你不再只是在写代码，你必须同时当指挥官与审查官。[2]

## 这本书写给谁（以及不写给谁）
这本书写给把产品当成一门手艺的人：愿意对结果端到端负责（价值、交付、治理），并且愿意用证据和门槛做决策。

这本书不写给两类读者：
- 只想找“魔法按钮”、不愿面对真实约束的人：AI 不能替你承担风险，它只会帮你更快制造技术债务。
- 只收集工具清单、却不建立工作纪律的人：0→1 的胜率来自节奏与裁决，而不是来自“装得更全”。[4]

## 你要对三件事负责（价值 / 交付 / 治理）
过去这些可能属于三个岗位；现在，它们都落到你一个人身上：

1. **价值**：先验证再建造。用最低成本证伪假设，明确反例与止损线；把“热闹”与“需求”区分开。[4]
2. **交付**：持续把改动变成可验收的成果。你要能说清“做到什么算完”，并让每次改动都能回归与回滚。[5]
3. **治理**：你对不确定性负责。AI 引入概率性，你必须把数据、评测与决策过程留档，让退化可解释、事故可追责。[34]

## AI 时代的最大风险：错误的确定性
AI 最危险的地方不是它会犯错，而是它会把错误包装得很确定：看起来合理、语气坚定、甚至还附带“最佳实践”。当实现成本变得很低，你会更容易把“能跑”误判为“成立”，把“偶尔成功”误判为“足够稳定”。[6]

因此，本书反复强调三件事：
- **证据**：没有证据的结论，默认只是猜测；猜测可以写，但必须附验证方法。
- **门槛**：门槛必须在改动前写下；改动后只按同口径对比表裁决。
- **回滚**：任何无法回滚的改动，都不值得在 0→1 阶段下注。

当你能用评测与回归把“感觉更好”变成“可裁决的提升”，AI 才真正成为你的杠杆。[18]

## 你将如何使用这本书（最短路径）
这不是一本“从第一页读到最后一页”的教材，更像一份作战手册。你可以按自己所处阶段取用：

- **你还在 0→1**：先跑通“机会 → 合同 → 原型 → 验证 → 体验”这条最短闭环：[`02-discovery.md`](02-discovery.md) → [`03-prd.md`](03-prd.md) → [`04-prototype.md`](04-prototype.md) → [`05-validation.md`](05-validation.md) → [`06-ui.md`](06-ui.md)。
- **你已经上线**：优先补齐“可回归”的基础设施：[`07-engineering.md`](07-engineering.md) 与 [`18-evaluation.md`](18-evaluation.md)；当你开始引入智能层，再进入 [`10-agent-rag.md`](10-agent-rag.md)。
- **你只想先抓骨架**：先读精华版：[`condensed-master-guide.md`](condensed-master-guide.md)。

## 写作承诺：克制但不空
我承诺不把本书写成“本周流行工具清单”。工具会过时，但方法与纪律更长寿。

我会尽量用最少的概念换取最大的推进：每章都围绕一条清晰的逻辑链条，提供可复制模板与验收门槛，并始终提醒你留下证据与退路。没有体系与纪律，“一人成军”只是口号；有了它，你才真的是一支队伍。[5]
