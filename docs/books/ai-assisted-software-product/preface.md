# 序言：一人即是一支队伍

![序言封面](../../assets/preface_header_1765970780362.png)

过去，要把一个脑中的想法变成能跑的产品，你需要一支整装待发的舰队：产品经理定航向、设计师画蓝图、前后端工程师造轮子、运维守底舱。

今天，AI 把“造轮子”的边际成本压缩到了接近零。这让“一个人成为一支队伍”不仅可能，而且正在发生。

但别搞错：这不是浪漫的自由，而是沉重的责任。

当代码不再是瓶颈，方向错误就成了最大的灾难。在低成本制造的诱惑下，你面临的最大风险不再是“做不出来”，而是“做出来一堆垃圾”：它们没有用户价值、无法持续交付，甚至因为缺乏治理而埋下法律或安全隐患。[4][5]

此刻起，你不再只是那个等着接需求的程序员。你必须同时佩戴三枚徽章：决断航向的**指挥官**（价值）、把改动变成成果的**工程师**（交付），以及在狂热中踩刹车的**审查官**（治理）。[2]

## 本书写给谁

这本书不教你“如何用 AI 一天生成 100 个垃圾站”，也不提供“快速暴富”的幻觉。这是一本写给严肃手艺人的工程手册。

- **你该读**：如果你想在不确定性里建立纪律，用证据而非直觉导航，并准备好对结果端到端负责（从写下一行字到收到第一块钱）。
- **别浪费时间**：如果你只把 AI 当作免思考的“外包工”，指望它替你承担思考的痛苦和决策的风险。

如果你愿意用“慢一点的验证”换取“稳一点的交付”，这本书提供的是一套把单兵作战变成可复利、可审计的工程体系。[4][5]

## 你的三重责任：价值、交付、治理

在一人成军的模式下，没人替你补位。你必须独自守住三条底线。

### 责任检查清单

| 角色 | 核心拷问 | 必须留下的可复核证据（Artifacts） | 失败判定（Stop Condition） |
| :--- | :--- | :--- | :--- |
| **价值（指挥官）** | 值不值得做？ | **决策日志**：包含假设、反例、止损红线 | 无法用一句话说清“谁在什么场景下解决什么痛点” |
| **交付（工程师）** | 做出来靠谱吗？ | **验收标准**：自然语言描述的“通过/失败”条件 | 自动化测试不通过；无法一键回滚 |
| **治理（审查官）** | 能长期存活吗？ | **评测报告**：包含准确率、幻觉率、延迟分布 | 关键动作无法回答“谁/何时/做了什么”；系统行为不可追溯 |

![图 P-1：价值/交付/治理三圈相交，中心为 AI Accelerator，并标注证据阈值、回滚阈值与审计阈值](../../assets/preface_p1_super_individual_1766373599659.png)

把上图看作你的三个“门神”：
- **价值门神**：没有可回放证据（原话/数据/对比表），不许写一行代码。
- **交付门神**：守门指标越界或关键 Case 失败，必须立刻降级或回滚，不许硬上线。
- **治理门神**：关键操作（例如扣费、删除数据）如果没有审计日志，不许扩量。

## 能力演进：从调用者到塑造者

这一路你会经历三个台阶。这不关乎职位，只关乎你能交付的确定性：

1. **Caller（调用者）**：你能把 API 调通，交付一个能跑的 Demo。局限：你不知道它为什么跑通，也不知道它什么时候会挂。
2. **Designer（设计者）**：你交付的是指标与护栏。你把“感觉更好”量化成“准确率 > 95%”和“延迟 < 200ms”。
3. **Shaper（塑造者）**：你交付的是权重与基准。你能通过调整数据分布与策略权重，让系统行为可控、可演进。

## 警惕 AI 的“虚假确定性”

AI 最危险的地方，在于它能以极其自信的语气胡说八道。当建造变得廉价时，你很容易产生幻觉：把“代码跑通了”误判为“路走对了”。[6]

我们必须建立一套无情的工程纪律：

- **证据（Evidence）**：没数据的观点叫猜测。猜测可以有，但必须写清验证方法与止损线。
- **门槛（Thresholds）**：把形容词换成数字。别说“要快”，要说“P99 延迟 < 500ms”。别说“要准”，要说“在固定评测集上 Pass Rate > 90%”。
- **回滚（Rollback）**：任何没有回滚按钮的功能，都是定时炸弹。在按下“发布”前，先问自己：如果炸了，我能在 10 分钟内恢复吗。

## 模板：现在就能用的工程脚手架

为了让你立刻上手，这里提供两个最基础的模板和一个可执行的门禁示例。

### 模板 A：一页纸 PRD（微型版）

在写代码前，强迫自己填完这个表格。填不出来就别动工。

| 字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| **用户故事** | As a <角色>, I want <功能>, So that <价值> | 作为主编，我想要一键校对，以便在 5 分钟内发现所有错别字 |
| **验收标准（AC）** | 具体的通过/失败条件 | 上传 Markdown 文件，返回带高亮的 HTML；错别字召回率 > 90%；误报率 < 5% |
| **止损线** | 投入多少资源未果即放弃 | 如果本周五前无法将误报率降到 10% 以下，则砍掉此功能 |

### 模板 B：决策日志（Decision Log）

新建一个 `DECISIONS.md`，像这样记录每一个关键转折。

```markdown
# 2025-12-24: 放弃本地部署 LLM，转用 API

- **背景**：本地模型响应太慢（P99 > 5s），且显存溢出导致 Crash。
- **决策**：切换到云端 API。
- **验证**：成本测算与性能测试结果写清（口径与样本固定）。
- **回滚方案**：保留本地推理方案，通过环境变量 `USE_LOCAL_LLM=true` 随时切回。
```

### 可执行示例：一个极简门禁脚本（可接入 CI）

这是一个极简 Bash 脚本，用来模拟“门神触发”。把它放在 `scripts/check_thresholds.sh`，并在 CI 或 pre-push hook 里调用。

```bash
#!/usr/bin/env bash
set -euo pipefail

echo "Running threshold checks"

# 1) 检查是否有硬编码密钥（示意规则）
if grep -R -nE "sk-[A-Za-z0-9]{20,}" ./src; then
  echo "FAIL: possible API key found in source. Move secrets to env vars."
  exit 1
fi

# 2) 运行一个最关键的冒烟测试（替换为你的实际用例）
python3 -m pytest tests/smoke_test.py

echo "PASS"
```

## 最短路径（今天就能做）

你不必读完本书才开始行动。以下 6 个动作，现在即可执行：

1. **新建 `DECISIONS.md`**：从今天起，别让决策只停留在脑子里。
2. **写下第一条止损线**：在你的待办事项旁写上“如果投入超过 `<X>` 小时仍未跑通，立即停止”。
3. **跑通空壳发布**：在功能开发完之前，先把 `Hello World` 的自动化部署流水线跑通。
4. **写一个金丝雀测试**：只写一个最关键的端到端用例，确保它永远是绿的。
5. **加一个回滚开关**：为新功能加一个 Feature Flag，一键关闭比现场修 Bug 快得多。
6. **清理幽灵代码**：删掉那些注释掉的代码块。Git 会替你记住，你不需要。
