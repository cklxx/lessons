# 第 2 章：需求挖掘与机会判断（Discovery）
![第 2 章封面](../../assets/chapter_02_header_1766035353733.png)

> 别急着写代码：先用数据把值不值得做说清楚，再用最小实验把它证伪/证实。[4]

!!! warning
    合规与成本提醒：先确认边界再动手。
    - **隐私/PII：** 评论常含个人信息。存储前必须脱敏；严禁将原始隐私数据直传公网模型或写入日志。
    - **平台条款：** 抓取前查阅目标站点服务条款与 `robots.txt`；优先使用官方 API 或导出功能。
    - **调用成本：** 大规模聚类/筛选优先使用本地/离线工具，将付费 LLM 算力集中于总结、反例挖掘与决策解释。

你可能正处于三种“虚假高效”之中：

1. **灵感过剩：** 每日点子频出，但缺乏事实支撑。
2. **数据堆积：** 抓取大量语料，结论模棱两可，无法指导决策。
3. **工具沉迷：** 精通聚类与 prompt，却不知何时停止分析。

本章目标是将重心从“产出报告”转向“做出决策”：所有结论必须溯源至证据并经过门槛检验，不达标即暂停或放弃。[4]

## 章节定位
旨在解决“热血开发两周却发现无人买单”的经典困境——将直觉转化为证据，将冲动转化为验证。[4]

本章核心任务：

1. **全信号采集：** 抓取公开样本，严格记录来源、时间与许可证，确保结论可复核、可用。[34]
2. **结构化压缩：** 将非结构化文本转化为《问题—证据》矩阵，包含样本量、反例及不确定性说明。[4]
3. **实验化裁决：** 设计 7 天验证闭环，设定明确通过门槛，不达标即止损。[4][6]

比起结论本身，本章更强调可复现性：通过去重、清洗模板文本与建立数据卡片（Datasheet），确保他人基于同源数据能得出近似结论。[34][35]

## 你将收获什么
- **《问题—证据》矩阵：** 替代主观观点，包含证据链、反例与下一步实验计划。[4]
- **7 天实验菜单：** 针对关键假设的验证方案，强调单变量与低成本。[4][6]
- **决策白板模板：** 明确“保留/放弃/延期”标准，设定止损红线。[4]

## 方法论速览
1. **假设可证伪：** 一句话定义“谁在什么场景下因何障碍遭受何种量化损失”。[4]
2. **信号即证据：** 多源采样 + 清洗去重 + 可追溯引用，将噪声压缩为评审素材。[34][35]
3. **寻找反例：** 主动挖掘不支持结论的场景，反例越早发现越节省成本。[4][10]
4. **门槛裁决：** 预设成功阈值与止损线，客观判定实验结果。[4][6]

## 把 Discovery 做成节奏，而不是阶段
很多人把需求挖掘当成一次性活动：前两周访谈、写一份报告、然后就开始冲刺开发。AI 时代这种做法更危险——因为你会更快地做出更多东西，也会更快地把错误方向做成看起来很像产品的样子。[4]

更稳的方式是把 Discovery 变成一种习惯：持续接触用户、持续提出可证伪假设、持续用最小实验裁决。它不是多开几场会，而是把证据纳入你的迭代节拍，让路线图永远和真实世界保持对齐。[73][72]

作图提示：画一棵树，从北极星指标/目标出发，向下分解成机会（用户痛点/场景），再分解成方案（原型/流程/能力），每个叶子挂一个实验卡片（门槛/失败判定/证据）。树上用颜色标出已验证/待验证/已否决。[73]

把机会→方案→实验写成一棵树，会强迫你在每一步都回答：这一步的证据是什么、下一步的门槛是什么、达不到就怎么停。[73]

```text
北极星指标/目标
  ├─ 机会：<用户痛点/场景>
  │    ├─ 方案：<最小能力/流程/原型>
  │    │    └─ 实验：<7 天内可证伪的门槛 + 失败判定 + 证据>
  │    └─ 方案：...
  └─ 机会：...
```

## 典型场景
- **SaaS 工具选题：** 抓 GitHub Issues、社区帖子、竞品评论，先回答用户到底在骂什么/夸什么。再决定做不做、先做哪个切入点。
- **B2C 应用验证：** 抽样应用商店评论，把卸载原因/留存动力做成可排序的清单，首版只做 Top 1–2 个关键问题。

## 实战路径
```text
数据抓取 → 去重/清洗 → 结构化压缩（证据绑定） → 主题/情绪建模 → 需求矩阵（问题—证据） → 虚拟访谈（反例） → 七天实验（门槛）
```

## 第一步：把问题写成可证伪假设（先别写需求列表）
你要的不是功能清单，而是假设清单。一个可证伪假设至少包含四要素：

> 对于【目标用户】，在【关键场景】下，他们因为【关键障碍】无法完成【关键任务】，导致【可量化损失】；如果我们提供【最小方案】，将把【关键指标】提升到【门槛】。

当你能把假设写到这一步，很多想法会自动缩水，因为它们无法被证伪，也无法被验收。[4]

## 第二步：信号采集（多源、可追溯、可复核）
采集的目的不是越多越好，而是足够代表你的目标人群。建议你为每个数据源补一份最小[数据卡片（Datasheet）](glossary.md#datasheet)：来源、时间窗口、采集方式、许可证、已知偏差与使用限制。[34]

**最低门槛：**
- 来源至少 2 个（否则你无法区分平台偏差和真实需求）。[34]
- 每条结论都能回指到原始样本（原文/链接/时间），否则视为不可用于决策。

为了让信号能更快被评审，可以让 AI 做结构化压缩，但必须强制证据绑定：每条记录都要引用原文中的一句话，不允许凭空总结。

```text
你是研究助理。任务：把下面的用户评论列表整理成 Markdown 表格。
输入：每行一条评论，已去重。
输出列：id | 主题 | 情绪(正/负/中) | 痛点 | 证据句(原文逐字引用) | 不确定性/反例 | 需要人工核验(是/否)
规则：
- 只允许基于输入文本；不得新增用户信息或背景。
- 每行必须给出证据句；没有证据句则标记需要人工核验=是。
- 无法判断写未知，不要编造。
```

## 第三步：《问题—证据》矩阵（把吐槽压缩成可评审表）
这张表是本章最重要的产物。它强迫你把观点拆成证据 + 反例 + 下一步实验。建议按下面模板填（先填 3 行就够了）：

| 痛点（问题） | 目标用户/场景 | 证据（原话/样本） | 样本量 | 反例/不确定性 | 可量化损失 | 下一步七天实验 |
| --- | --- | --- | ---: | --- | --- | --- |
|  |  | 至少 3 条引用 |  |  |  |  |

!!! note
    作图提示：建议画成一张可以开评审会的表，行列规则先定死。
    - 行：一个痛点/一个假设（不要把多个问题揉成一行）
    - 列：证据（原话）/样本量/反例/可量化损失/七天实验（把下一步写成可执行动作）
    - 用颜色或标签标出：高风险（合规/越权/成本）与高价值（高频/高痛/高愿付出）

**填表示例（示意，非你的真实数据）：**

| 痛点（问题） | 目标用户/场景 | 证据（原话/样本） | 样本量 | 反例/不确定性 | 可量化损失 | 下一步七天实验 |
| --- | --- | --- | ---: | --- | --- | --- |
| 导入太麻烦，第一天就流失 | 新用户/首次使用 | 导入卡在第三步、不知道该传什么格式… | 27 | 部分用户不需要导入也能用 | 首次任务完成率低 | Concierge：手把手导入 + 记录耗时与失败点 |

这张表的价值在于：它让评审不再讨论你感觉对不对，而是讨论证据是否足够、反例是否被覆盖、实验门槛是否合理。[4]

## 从痛点到价值主张：用一句话把产品钉住
做完《问题—证据》矩阵后，你最容易犯的一个错误是：继续沉迷于更完整的分析，却迟迟不敢把产品承诺写出来。可在 0→1 阶段，价值主张越晚写出来，你越容易把所有痛点都想解决误判为产品更强。

更稳的做法是：从矩阵里挑 1 条最能代表价值闭环的痛点，把它改写成一句能贴到登陆页、也能贴到 PRD 的承诺句。它不需要漂亮，但必须可证伪：

- **对象**：为谁（不要写所有人）
- **场景**：在什么时候（触发条件）
- **结果**：得到什么可感知收益（不是更智能）
- **代价**：用户要付出什么（数据/迁移/学习成本）

你会发现，一旦把这句话写出来，很多看似不错的功能会自动被排除：它们无法缩短闭环、无法降低不确定性、也无法在七天内被验证。[4]

## 第四步：反例优先（用访谈把方向拉回地面）
你最需要的不是更多赞同，而是更快被打脸。建议你把访谈当成反例探测器：

- **替代方案**：你现在怎么解决？为什么还愿意忍？
- **付出成本**：如果要试用，你愿意付出什么？（时间/数据/钱/迁移）
- **触发条件**：什么情况你一定不用？什么情况你一定会用？

如果你使用 [Agent（智能体）](glossary.md#agent) 做虚拟访谈，它的角色也应该是反例生成器，而不是帮你证明结论。[10]

!!! warning
    虚拟访谈不是证据。它只能用来生成反例与提问清单，不能生成用户说过什么或用户会怎么做的结论。任何来自虚拟访谈的结论，都必须回到真实访谈/真实行为数据中回证，否则不得进入《决策白板》。

```text
你是反例生成器。输入：一个可证伪假设 + 当前证据摘要。
输出：
1) 10 个可能让假设不成立的反例场景（每条都写：如何验证）。
2) 10 个访谈追问问题（每题都写：想排除哪类误判）。
硬性规则：不得声称已访谈；不得编造用户原话；不确定就写未知。
```

## 第五步：七天实验菜单（用门槛裁决，而不是用热情）
0→1 阶段最容易死在永远在验证，却从不做决定。你需要把实验做得足够短、足够狠：提前写清门槛与止损线。[4][6]

| 实验类型 | 适用场景 | 你要测什么 | AI 加速点（例） | 成功门槛（示例，需按基线校准） | 失败判定 | 典型成本 |
| --- | --- | --- | --- | --- | --- | --- |
| 登陆页 + 候补名单 | 价值主张不清 | 付出意愿 | 生成 3 版文案与 FAQ 变体 | 转化率 ≥ 基线 × 1.5 | 低于基线 × 1.2 | 低 |
| Concierge / Wizard-of-Oz | 流程复杂/需人工验证 | 闭环是否发生 | 自动整理卡点与问题清单 | 关键任务完成率 ≥ 基线 + 20pp | 关键步骤反复卡死 | 中 |
| 可点击原型测试 | 信息架构/交互不确定 | 可发现性/错误恢复 | 快速产 2–3 套原型文案/空状态 | 任务完成率 ≥ 基线 + 15pp | 错误恢复失败率高 | 中 |
| 最小可用版本（小流量） | 方案已清楚 | 留存/复用 | 辅助生成埋点/回归用例/触达文案 | 7 日留存 ≥ 基线 + 3pp | 留存低且无反例解释 | 高 |

你不需要在第一周就追求统计显著，你需要的是可复现趋势 + 下一步如何证伪。[4][6]

## 第六步：决策白板（把结果落到保留/放弃/延期）
每周/每两周输出一页决策白板，强迫自己做选择：[4]

| 假设 | 证据 | 反例 | 实验结果 | 决策（保留/放弃/延期） | 下一步 | 止损线 |
| --- | --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |  |

!!! note
    作图提示：建议画成一张每两周必填的白板，强迫自己做裁决。
    - 左侧：本轮要验证的 1–3 个假设（对应矩阵中的高优先级行）
    - 中部：证据与反例（只允许写可回放的证据：原话/录屏/日志摘要）
    - 右侧：裁决（保留/放弃/延期）+ 下一步实验 + 止损线（把继续努力改成继续验证什么）

## 关键流程图（纯文本）：Discovery 一周节拍

```text
多源信号采集（可追溯） -> 数据卡（许可/时间窗/偏差） -> 去重/清洗（可复跑）
  -> 《问题—证据》矩阵（证据/反例齐全） -> 反例优先访谈（拉回地面）
    -> 七天实验（门槛+止损线） -> 决策白板（保留/放弃/延期） -> 下一步或暂停

门禁（任一不满足就停止产出结论）：
- 数据不可追溯或许可不清 -> 先补数据卡与采样策略
- 痛点无证据或无反例 -> 先补样本与反例访谈
- 实验无门槛/无止损线 -> 先补裁决条件，再允许继续做
```

## 示例（可复制）：把吐槽变成决策白板（7 天）

**目标：** 7 天内对一个方向做裁决（保留/放弃/延期），并留下可复盘证据。

**前置条件：**
- 至少两类信号源（例如工单 + 商店评论 / 论坛帖子 + 用户访谈摘录）
- 你有一个可提交的证据目录（例如 `docs/discovery/`）与一个可归档的报告目录（例如 `reports/`）

**约束：**
- 不提交原始 PII；样本摘录必须脱敏。
- 数据许可与时间窗必须写清；许可不清的来源不得支撑结论。[34][35]

**输出格式：**
- 数据卡：`docs/discovery/datasheet.md`
- 《问题—证据》矩阵：`docs/discovery/problem-evidence-matrix.md`
- 七天实验卡：`docs/discovery/exp-7d.md`
- 决策白板：`docs/decisions/dec-YYYY-MM-DD.md`

**步骤：**
1. 把两类信号源写进数据卡：来源、时间窗、采样偏差、去重/过滤规则、抽检样本（至少 20 条）。[34][35]
2. 用《问题—证据》矩阵压缩讨论：每条痛点至少 3 条可回放证据 + 1 条反例，缺任一项不得进入路线图。[4]
3. 从矩阵选 1–3 个高优先级假设，写七天实验：指标口径、门槛、止损线、失败恢复（达不到就停）。[4][6]
4. 跑完七天后写决策白板：保留/放弃/延期，并写清下一步与止损线。[4]

**验证命令：**
```bash
python3 - <<'PY'
from pathlib import Path

required = {
  'docs/discovery/datasheet.md': ['来源', '时间', '许可证'],
  'docs/discovery/problem-evidence-matrix.md': ['问题', '证据', '反例'],
  'docs/discovery/exp-7d.md': ['门槛', '止损', '失败判定'],
}

missing = [p for p in required if not Path(p).exists()]
if missing:
  raise SystemExit(f'missing files: {missing}')

for path, must in required.items():
  text = Path(path).read_text(encoding='utf-8', errors='replace')
  for k in must:
    if k not in text:
      raise SystemExit(f"missing '{k}' in {path}")

print('ok')
PY
```

**失败判定：**
- 任何结论无法回指到样本/证据，或无法解释反例。
- 七天实验没有门槛与止损线，或指标口径写不清。

**回滚：**
- 将该方向裁决为延期/放弃，并把原因写进决策白板；下一轮只允许从补齐数据与口径开始，而不是继续扩功能。[6]

## 复现检查清单（本章最低门槛）
- 数据可追溯：来源、时间窗口、许可证、去重/过滤规则与采样偏差写清楚，并能回到原始样本（至少抽检 20 条）。[34][35]
- 结论可评审：《问题—证据》矩阵每条都有样本量、原话引用与反例；缺任一项视为无效，不得进入路线图。[4]
- 决策可裁决：每个高优先级假设都有七天实验、门槛与止损线；达不到就暂停/放弃，并记录原因与下一步。[4][6]
- 流水线可复跑：采集→清洗→提炼→矩阵→白板能在同一数据快照下复跑；若结论不稳定，必须能解释差异来自哪里（随机种子/清洗参数/采样偏差）。[9][26]

## 常见陷阱（失败样本）
1. **现象：** Top 痛点看起来很清晰，但上线后没人用。  
   **根因：** 采样只来自单一平台/单一人群，你听到的可能只是声量最大的少数群体；真正的沉默的大多数直接卸载、从不评论。[34]  
   **复现：** 只用 App Store/只用某论坛的评论跑一次主题建模，对比不同渠道 Top5 主题差异。  
   **修复：** 跨源采样并记录权重与时间窗口；在矩阵里为每个痛点标注来自哪些源/占比。[34]  
   **回归验证：** 多源数据的 Top 主题在同口径复跑下仍稳定（或明确写出分歧与取舍）。

2. **现象：** 主题模型每次重跑结论都变，评审无法对齐。  
   **根因：** 去重/清洗与随机性控制缺失，重复与模板化文本导致主题漂移。[35]  
   **复现：** 同一份数据在不同清洗参数/不同随机种子下重跑，观察主题排序变化。  
   **修复：** 固定数据快照、清洗参数与随机种子；对重复文本做 MinHash/LSH 去重，并记录差分。[35]  
   **回归验证：** 抽检 3 次运行结果，Top5 主题的覆盖度与样本量变化不超过阈值（阈值由你的基线决定）。[9]

3. **现象：** 团队每周都在验证，但从未做出可执行的决策。  
   **根因：** 假设过多、实验不可证伪、没有停止条件，导致验证永远在路上。[4]  
   **复现：** 罗列当前所有假设，发现多数无法在七天内证伪或没有明确指标。  
   **修复：** 每周最多 3 个待验证假设；每个假设必须绑定证据/反例/七天实验/门槛/回滚。[4][6]  
   **回归验证：** 每两周产出一页《决策白板》（保存在版本库），明确保留/放弃/延期的决策与证据链。[4]

## 延伸练习
- 将痛点主题映射到现有竞品的功能列表，验证市场空白是否真实存在。
- 把你的七天实验复跑两次：对比两批相似人群的方向是否一致，避免一次性结论带偏方向。

## 交付物清单与验收标准
- 数据卡片（Datasheet）：来源/时间窗口/许可证/偏差说明/去重与过滤规则。[34]
- 《问题—证据》矩阵：每条痛点至少 3 条引用 + 样本量 + 反例 + 下一步实验。[4]
- 《决策白板》：每两周至少一次保留/放弃/延期的裁决记录，含止损线。[4]

下面把本章的实战路径抽象为可迁移的原则，便于你换数据源/工具链时仍能复现与审计。

## 深度解析：核心原则
1. **场景化起手式：** 从一条用户投诉开始——用正反两种假设框定问题，再让 [Agent（智能体）](glossary.md#agent) 爬取 1,000 条相似评论验证是否具备共性痛点。以混合检索（例如 BM25 + 向量相似度）定位语义近邻，拒绝凭感觉扩写需求。[7][27]
2. **数据护栏：** 每个数据源附带可信度评分（采集渠道、样本量、时间跨度、噪声水平），分为 A/B/C 级。低于 B 级的数据仅做辅助，不得支撑产品决策，并在数据卡片中明示原因；对重复与模板化文本应明确去重策略。[34][35]
3. **主题—用户—价值对齐表：** 构建三维表格：主题（痛点）、目标 Persona、价值/节省的成本或时间。每行附来源与样本量，未能量化价值的主题暂缓进入路线图。[8]
4. **一页式决策白板：** 每周输出一次决策白板，包含当前假设、证据、反例、风险、下周实验。建议在你的项目仓库中以编号保存（示例：`docs/decisions/`），作为后续复盘的审计链路。[4]
5. **复现流水线：** 把采集→清洗→提炼→矩阵→决策白板做成一条可重复的任务，并把关键口径（样本量、Top 主题覆盖度）写进报告。失败就停止产出结论，先回到数据与口径层面修正。[9][26]
