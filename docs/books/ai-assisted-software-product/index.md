# 写给未来的超级个体：AI 全栈产品开发实战

> 面向超级个体的实战手册：覆盖从机会发现、工程交付到模型训练与商业闭环的全链路。本书只强调一件事：**可裁决性**——关键论点尽量给出权威来源或**验收标准**（对比表、评测集、决策日志）。注意：本仓库是方法论文档，不绑定具体技术栈；重点在于建立同口径的证据链与回滚机制。[1][2][5][18]

## 范围与前置假设

- **适用对象**：个人开发者/小团队；希望用 AI 提升验证速度、工程交付与模型能力，并最终跑通商业闭环的人。
- **默认能力基线**：你能独立开发并部署一个小型 Web 产品；能阅读并修改 Python/TypeScript 代码；对基本统计概念（抽样、偏差、显著性）不陌生。[4][18]
- **资源约束说明**：训练/推理章节默认你具备可租用 GPU 的能力；如果你只有 CPU 或轻量 GPU，建议优先走量化、LoRA/QLoRA、蒸馏或托管推理等低成本路径——先把闭环跑通，再谈重武器。[44][47]

## 可复现性与断言的定义

- **可复现**：在明确输入、环境与随机性控制（种子/版本/数据快照）的前提下，读者能够复跑并得到同级别结论（同趋势、同阈值判断），而不是要求逐 bit 相同输出。[1][5]
- **断言**：必须可被证伪。若无法通过实验/日志/评测指标验证，应明确标注为经验性建议或开放问题，并给出验证方法。[18]

## 本书的写法（你会看到的风格）
我尽量把这本书写成一份可长期复用的作战手册，而不是工具清单：

1. **范式先行，工具靠后**：先讲可迁移的方法与结构，再把工具当作可选实现路径，避免把某个版本当成结论。[2][24][29]
2. **每章以裁决为终点**：不追求说服，追求可裁决——门槛写清、证据留档、退化能回滚。[5][18]
3. **模板集中在附录**：正文尽量讲清为什么与怎么想；可直接复制的模板/提示词/清单统一收敛到附录，方便你照抄、裁剪、版本化。[18]
4. **格式为结构服务**：优先用标题/列表/表格表达层次，用完整句子解释因果；只对命令/路径/字段名/字面量用代码格式；正文避免用英文双引号（U+0022、U+201C、U+201D）给术语加戏；模板允许少量占位符，但要统一格式并说明替换点。更完整的约定见 [写作风格与格式约定](style-guide.md)。

## 使用方式与质量门槛
- **验证闭环**：每章提供实操与验证小节，强调**验证方法**与**可复现证据**（日志/评测集/基准表/截图），并给出如何把这些检查自动化进 CI 的建议。[5][28][44]
- **变更对比**：涉及检索、训练、部署等关键配置时，推荐提交改动前/后的指标对比（延迟、质量、成本、风险面）；没有证据支撑的优化应回到假设与实验设计层面重做。[5][28][44]
- **资产版本化**：把数据快照、评测集、Notebook、配置与报告一起版本化，形成可审计的个人知识资产；对应统一门槛参见《全书质量控制与复现清单》。[1][34]

## 致读者
本书旨在帮助个人开发者与小团队，以最少的资源完成从需求验证、全栈实现到模型训练与部署的闭环。请在阅读时结合自身约束（时间、算力、预算），灵活采纳各章节的低成本路径。
