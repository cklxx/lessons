# 写给未来的超级个体：AI 全栈产品开发实战

> 面向“超级个体”路线设计，从机会发现到上线运营、模型训练与商业闭环的全链路方法手册。本书尽量为关键观点提供权威来源（论文/标准/经典著作），并给出可操作的复现路径（最小示例 + 检查清单）。注意：本仓库是文档站点，不内置完整工程代码；文中出现的 `make ...` / `CI` / 目录结构是“建议如何落地到你自己的项目仓库”的示例约定，而非本仓库现成可执行脚本。[1][2][5][18]

## 范围与前置假设

- **适用对象**：个人开发者/小团队；希望用 AI 提升“验证速度 + 工程交付 + 模型能力”的端到端闭环。
- **默认能力基线**：能独立完成一个小型 Web 产品的开发与部署；能阅读并修改 Python/TypeScript 代码；理解基本的统计概念（抽样、偏差、显著性）。[4][18]
- **资源约束说明**：训练/推理章节默认你具备可租用 GPU 的能力；若只有 CPU/轻量 GPU，请优先采用量化、LoRA/QLoRA、蒸馏或托管推理服务等低成本路径。[44][47]
- **内容组织说明**：`chapter-*.md` 为正文；同目录下以数字开头的 `01-*.md` 等文件是写作过程的工作底稿/提示词记录，默认不参与站点构建与导航（避免与正文重复或冲突）。

## 可复现性与“断言”的定义

- **“可复现”**：在明确输入、环境与随机性控制（种子/版本/数据快照）的前提下，读者能够复跑并得到同级别结论（同趋势、同阈值判断），而不是要求逐 bit 相同输出。[1][5]
- **“断言”**：必须可被证伪。若无法通过实验/日志/评测指标验证，应明确标注为经验性建议或开放问题，并给出验证方法。[18]

## 写作准则（The Constitution）
> 在撰写任意章节前需满足的五条“护栏”，用于质量控制与对齐“超级个体”定位。

1. **行动优先（Action Over Theory）**：每节至少提供一种“可操作输出”（最小可运行示例/可复用 Prompt/可直接落地的配置片段）。如果示例依赖外部生态或未给出完整上下文，必须显式标注为“伪代码/片段”，并说明前置条件与替代实现。[18]
2. **成本意识（ROI Mindset）**：以个人/小团队为视角优先低算力与低成本方案（量化、LoRA/QLoRA、开源推理引擎、按需租用算力），并清楚写出成本估算的假设与区间，而不是给绝对数字。[44][47]
3. **工程化落地（Production Grade）**：凡是“可上线”的建议，至少说明鉴权/错误处理/日志/并发/回滚策略的最小集合；不把一次性 Demo 当作生产方案。[1][5]
4. **范式优先（Patterns Over Tools）**：先讲可迁移的范式（Transformer、RAG、Agent、评测与治理），再给工具清单作为“可选实现”；避免把工具版本当作结论本身。[2][24][29]
5. **端到端负责（End-to-End Ownership）**：每章都明确“产品指标 ↔ 工程决策 ↔ 成本/风险”的映射关系，解释为什么这个技术选择会影响转化、留存或合规风险。[4][6][23]

## 使用方式与质量门槛
- 每章提供“实操与验证”小节：强调**验证方法**与**可复现证据**（日志/评测集/基准表/截图），并给出如何把这些检查自动化进你自己的 CI 的建议。
- 涉及检索、训练、部署等关键配置时，推荐提交“改动前/后”的指标对比（延迟、质量、成本、风险面）；没有证据支撑的“优化”应回到假设与实验设计层面重做。[5][28][44]
- 建议把数据快照、评测集、Notebook、配置与报告一起版本化，形成可审计的个人知识资产；对应的统一门槛参见《全书质量控制与复现清单》。[1][34]

## 致读者
本书旨在帮助个人开发者与小团队，以最少的资源完成从需求验证、全栈实现到模型训练与部署的闭环。请在阅读时结合自身约束（时间、算力、预算），灵活采纳各章节的“低成本路径”。
