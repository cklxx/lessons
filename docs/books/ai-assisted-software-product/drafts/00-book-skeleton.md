# 去工具化重写骨架（作者备忘）

> 目标：把工具手册感降下来，把作者的判断力与叙事写出来；正文只保留读者真正需要的因果、门槛与退路，把提示词/模板/作图想法集中到附录与 drafts。

## 1) 写作立场：像作者，而不是像工具箱

- **从读者真实处境起笔**：读者不是缺工具，而是缺我现在该先做什么、为什么这么做、做到什么算完、没做到怎么办的判断链。
- **不写谁不适合**：把排除式声明改成提醒式约定。用我希望你尽量别做 X，因为 Y 会发生，替代本书不写给 X。
- **克制但不单薄**：每个关键观点至少补齐三件事：场景（何时会遇到）→ 原因（为什么会这样）→ 动作（怎么做/怎么验收/怎么回滚）。

## 2) 去工具化策略：正文讲范式，工具留在附录

- **正文不堆工具清单**：工具名最多作为可选实现出现，避免让读者误解为选对工具就等于选对方向。
- **模板与提示词收敛**：把可复制的内容集中到 `A-templates.md`、`B-prompts.md`、`C-checklists.md`，正文只保留 1 个代表性示例并回链。
- **移除作者占位内容**：正文不出现作图提示、占位图、编辑备忘；这些材料留在 `drafts/`，方便后续补图与修订。

## 3) 全书主线：三件事负责（价值 / 交付 / 治理）

- **价值**：先判断值不值得做（证据与止损），再把愿景写成合同（验收与回滚），再用原型与验证让闭环发生。
- **交付**：把能跑变成能上线：回归、门禁、可观测、可回滚。
- **治理**：把偶尔很聪明变成长期可托付：证据链、边界、评测门禁、审计与合规。

## 4) 章节骨架（每章一句话 + 读者要拿走的产物）

- `preface.md`：把一人成军的浪漫写成三座灯塔（价值/交付/治理），并立下证据、门槛、回滚的写作承诺。
- `01-method.md`：给读者一条端到端管线，让 AI 的速度进入纪律（而不是进入发散）。
- `02-discovery.md`：把热闹变证据，把证据变实验，把实验变裁决（保留/放弃/延期）。
- `03-prd.md`：把 PRD 写成合同：范围、异常流、NFR、验收、回滚都要可裁决。
- `04-prototype.md`：用最小原型验证路径能走通、失败能回来，尤其把 AI 的不确定性画进体验里。
- `05-validation.md`：把打磨变成一周节拍：迭代卡片 → 对比表 → 裁决 → 归档。
- `06-ui.md`：把好看写成可维护资产：设计系统、a11y、状态与一致性。
- `07-engineering.md`：把 AI 纳入交付流水线：约束先行、检查裁决、灰度与回滚。
- `08-frontend.md`：把体验写成可预测系统：状态矩阵、错误可恢复、进度/证据/反馈三件套。
- `09-backend.md`：把系统写成事实源：边界、语义、幂等、审计、可观测与预算。
- `10-agent-rag.md`：智能层不是能跑，而是有证据、有边界、可回归。
- `11-user.md`：用户模块不是登录，而是权限边界与审计证据链。
- `12-billing.md`：计费不是扣款，而是不可变账本、对账闭环与止损线。
- `13-data.md`：数据不是堆语料，而是可追溯、可脱敏、可版本化的资产。
- `14-pretrain.md`：预训练以增量适配思路写：先便宜方案，再训练；双基线评测，不退化才算赢。
- `15-posttrain-rl.md`：后训练以对齐与门禁写：偏好数据、攻击回归、灰度与回滚。
- `16-inference.md`：推理优化不是更快，而是质量/延迟/成本三角的可裁决取舍与降级。
- `17-deployment.md`：部署与运维写成可控发布流程：版本化、灰度、观测、回滚、复盘。
- `18-evaluation.md`：评测体系写成日常治理：离线回归、在线观测、红队、门禁与回滚。
- `19-iteration.md`：增长写成实验纪律：路线图、实验卡片、裁决与资产化。
- `20-governance.md`：合规与伦理不是提醒，而是边界、审计、处置与复盘的系统能力。
- `conclusion.md`：收束到长期可托付的标准：不是速度，而是可解释、可回归、可止损。
