# 第 3 章：PRD 书写：从愿景到验收

## 本章目标

- 写出能驱动设计与研发落地的 PRD（不是需求作文）
- 用 AI 扩展边界条件与用例，但用验收标准收敛范围
- 把 PRD 变成全团队的合同：目标一致、争议可追溯

## PRD 推荐结构（大纲）

### 3.1 背景与问题陈述

- 现状、痛点证据、为什么现在做

### 3.2 目标 / 非目标

- 业务目标（可量化）与产品目标（用户价值）
- 非目标：明确不做什么，防止范围膨胀

### 3.3 用户与场景

- Persona、关键旅程、边缘人群

### 3.4 需求范围：用户故事与用例

- 用户故事（As a / I want / so that）
- 用例流：主流程、备选流、异常流

### 3.5 功能需求（Functional Requirements）

- 模块拆分：信息架构、页面与组件、API 与数据

### 3.6 非功能需求（NFR）

- 性能、可用性、可观测性、安全、隐私、合规、成本预算

### 3.7 指标与埋点

- 北极星指标、过程指标、质量指标（延迟/失败率/成本）

### 3.8 验收标准（Acceptance Criteria）

- 用 Gherkin（Given/When/Then）或清单式描述
- 与测试用例、监控指标和上线回归绑定

### 3.9 风险与开放问题

- 风险清单：技术、数据、合规、依赖
- 需要拍板的决策点（Decision Log）

## 本章交付物

- PRD（含验收标准、NFR 与风险清单）
- 需求到测试的追踪矩阵（Requirement Traceability Matrix）
