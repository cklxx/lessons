## TL;DR
Prompt 控件（Prompt Controls）是位于 AI 聊天输入框周边的辅助 UI 组件，旨在通过混合界面（Hybrid UI）降低用户的交互成本；其核心价值在于无需用户“猜功能”，直接显性化地提供功能发现、灵感启发、对话约束及追问引导。

## 核心观点

1.  **混合 UI 是必然趋势**：纯文本对话界面（LUI）虽然灵活，但交互成本高。Prompt 控件（按钮、菜单、Chips）与文本框结合的“混合 UI”能有效解决“不知道能问什么”和“不知道怎么问”的问题。
2.  **打破“表达障碍”**：用户往往难以用精准的 Prompt 描述需求。控件通过提供预设选项（如“聚焦搜索范围”、“生成图表”），替用户完成了复杂的上下文设定。
3.  **四大核心场景**：
    *   **发现功能**：避免用户用自然语言去试探“你能读 PDF 吗？”，直接展示上传图标或格式提示。
    *   **教育与灵感**：利用“对话启动器（Conversation Starters）”和 Prompt 库，教新用户如何使用 AI。
    *   **设定约束**：在对话开始前划定范围（如 Perplexity 的“Focus”功能，限定搜索 Reddit 或学术论文），或指定输出格式（如“生成思维导图”）。
    *   **简化追问**：77% 的对话包含多轮交互。控件应支持“一键修改”（变短、变正式）、引用回复内容（Apple-picking）等高频后续操作。
4.  **图标必须带标签**：AI 功能通常较新，用户缺乏既定心智模型。孤立的图标（如“魔法棒”）极易造成困惑，必须搭配文字标签或悬停提示（Tooltip）。
5.  **拒绝“品牌黑话”**：功能命名应直观描述作用（如“文字转视频”），而非使用晦涩的内部品牌词（如“一键流影”），以免增加用户的认知负荷。
6.  **遵循 Jakob 定律**：不要随意劫持用户习惯的标准交互手势。例如，长按消息应弹出标准的复制/选择菜单，而非强制执行特定 AI 操作。

## 可落地做法

**面向产品经理（PM）：**
*   **功能显性化盘点**：梳理产品中哪些能力是用户高频询问“你能做吗？”的，将其转化为显性的 UI 控件（如文件上传支持的格式提示）。
*   **设计“中间层”交互**：在 Prompt 输入前和输出后，增加非文本的交互层。输入前允许配置参数（约束），输出后允许快捷指令（追问）。

**面向设计师（UI/UX）：**
*   **建立控件分组**：依据功能逻辑对插件和工具进行分组（Gestalt 邻近原则）。不要把“文件上传”（上下文）和“生成视频”（输出格式）混在一个杂乱的菜单里。
*   **错误反馈就近原则**：错误提示（如文件过大）应直接出现在触发错误的控件（上传按钮）附近，而非屏幕顶部容易被忽略的非模态弹窗。

**面向工程研发（Dev）：**
*   **引用回复（Quoting）支持**：实现富文本编辑器的引用功能，允许用户高亮选中文本片段后，点击“引用”按钮将其自动填入输入框，方便针对性追问。

## 检查清单：Prompt 控件设计审计

可用于 UI 走查或 PRD 评审：

*   **可发现性与功能明确性**
    *   [ ] 输入框周边的所有图标是否都配有文字标签或悬停 Tooltip？
    *   [ ] 图标语义是否标准？（如：避免用“扫把”代表“新话题”，除非有强引导）
    *   [ ] 功能命名是否直观？（拒绝品牌造词，如用“深度思考”代替不明所以的代号）
    *   [ ] 是否在用户未输入前展示了“对话启动器”或示例 Prompt？
*   **交互逻辑与反馈**
    *   [ ] 长按、右键等标准手势是否保留了系统默认行为（或包含默认行为）？
    *   [ ] 错误提示是否在视觉上紧邻触发错误的控件？
    *   [ ] 点击控件后，是否有明确的状态变化（如：选中态、加载中）？
*   **信息架构**
    *   [ ] 功能相似的控件是否分在一组？（如：输入辅助类 vs 输出控制类）
    *   [ ] 控件数量是否过多导致认知过载？（是否需要折叠或分级）
    *   [ ] 是否提供了便捷的“追问/修改”按钮（如：重试、复制、编辑上一条）？

## 常见坑与对策

| 常见坑 (Bad Practice) | 后果 | 对策 (Countermeasure) | 案例反面/正面 |
| :--- | :--- | :--- | :--- |
| **神秘图标** | 用户看不懂“魔法棒”是做图还是润色，不敢点。 | **图标+标签**。若空间有限，至少确保 Hover 有 Tooltip。 | ❌ 百度文心一言 App 的魔法棒图标让人困惑。<br>✅ ChatGLM 底部图标带 Tooltip。 |
| **品牌化自嗨命名** | 用户不知道“灵感火花”是写代码还是写诗。 | **功能性命名**。直接叫“代码生成”或“文案润色”。 | ❌ 某插件叫“一键流影”（含糊）。<br>✅ 改名为“文字转视频”。 |
| **劫持标准手势** | 长按文字想复制，结果自动触发了 AI 解析。 | **尊重原生交互**。在原生菜单中“追加”AI 选项，而非“覆盖”。 | ❌ ChatGLM App 长按强制复制整段。<br>✅ ChatGPT App 长按弹出菜单含选择/复制/编辑。 |
| **大杂烩菜单** | 上传文件、联网搜索、画图全塞在一个“+”号里。 | **逻辑分组**。将“上下文输入”与“能力开关”分离。 | ❌ 早期文心一言插件中心杂乱。<br>✅ Perplexity 将“Focus”（范围）与“Attach”（附件）分离。 |
| **错误提示离家出走** | 底部上传失败，顶部飘过一行小字。 | **就近反馈**。在操作点附近提示错误。 | ❌ 底部上传，顶部非模态报错。<br>✅ 输入框下方直接红字提示。 |

## 可用于丰富《AI 辅助软件产品》的写作点

*   **第 04 章 Prototype（原型设计）**：
    *   在设计 AI 产品原型时，**严禁只画一个聊天气泡和输入框**。
    *   引用本文观点，强调“混合 UI”是原型设计的基准，必须规划输入框周边的功能区（上传区、Prompt 库、模式切换）。
    *   **案例**：对比 ChatGPT 早期版本与现版本的输入框演变，说明 Prompt 控件如何逐步丰富。

*   **第 06 章 UI（用户界面）**：
    *   **Prompt Engineering 的 UI 化**：将复杂的 Prompt 技巧（如 Few-shot, Chain-of-Thought）封装为 UI 控件。例如，Perplexity 的“Focus”本质上是在 System Prompt 中注入了特定领域的约束，但在 UI 上表现为简单的按钮。
    *   **交互惯例**：专门有一节讨论“AI 原生交互 vs 传统交互习惯”，强调 Jakob 定律在 AI 时代依然适用（如不要劫持长按）。

*   **第 11 章 User（用户体验与引导）**：
    *   **新用户引导（Onboarding）**：利用 Prompt 控件解决“冷启动”问题。
    *   **降低认知负荷**：引用 NN/g 的数据（“19% 美国用户从未听说过 ChatGPT”），说明大部分用户是 AI 小白，必须通过显性的 UI 控件（如对话启动器）来教育用户 AI 的能力边界。
    *   **Apple-picking 模式**：介绍用户如何通过“引用（Quoting）”来从 AI 的长回复中“摘取”需要的部分进行迭代，这是人机协作的高频模式。
