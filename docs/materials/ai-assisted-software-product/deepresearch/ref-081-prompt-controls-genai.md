## TL;DR
Prompt Controls 将隐性的提示词指令转化为显性的 UI 组件，通过按钮、菜单和开关降低用户的交互成本与认知负担，从而提升 AI 系统的可用性与功能发现率。

## 核心观点
1. 混合界面优于纯文本交互，通过 UI 控件补充输入框，帮助用户克服表达障碍
2. 控件的首要价值是提升可发现性，让用户直观看到系统支持文件上传或特定格式输出，无需试探
3. 预设指令具有教育意义，对话启动器和灵感中心能教导新用户系统的能力边界
4. 约束前置化，允许用户在对话开始前通过 UI 锁定搜索范围、输出格式或专注领域
5. 追问操作产品化，将缩短、扩写、改变语气等高频修改需求做成点击操作，促进多轮对话
6. 标签必须清晰明确，避免使用晦涩的品牌词或无文字说明的图标，降低理解成本
7. 控件分组应符合逻辑，避免将上下文设置、输出格式和业务插件混杂在一起
8. 遵循现有交互惯例，不应为了创新而破坏用户对长按、点击等基础操作的肌肉记忆

## 可落地做法
1. 产品设计侧
梳理用户对话日志中的高频意图，将前 20% 的通用指令（如总结、润色、代码解释）转化为快捷操作区。
在输入框周边增加能力提示，例如鼠标悬停上传图标时显示支持的文件格式与大小限制。
设计输出后的操作栏，提供 复制、重试、评价 以及 缩短、更专业 等修改维度的快捷入口。

2. 前端工程侧
建立 Prompt Controls 组件库，包含 胶囊按钮、快速命令菜单、引用浮层 等标准组件。
实现控件与输入框的联动，例如点击控件自动填充提示词，或直接发送带参数的请求。
处理移动端的交互冲突，确保长按引用或复制的体验与系统原生行为一致。

3. 质量评测侧
测试控件触发的 Prompt 模板在不同模型下的稳定性，确保 点击更短 确实能产生 Tokens 更少的结果。
验证控件状态与后端逻辑的一致性，防止 UI 上选了 仅搜索学术界 但后端仍检索全网。

## 检查清单
UI 控件审查表
1. 所有非通用图标是否都配有文字标签或悬停提示
2. 功能名称是否直白描述了作用（如 视频生成 而非 一键流影）
3. 是否在空状态下提供了 3 至 5 个对话启动示例
4. 上下文设置（如联网开关）与输出控制（如生成图表）是否在视觉上做了区分
5. 报错信息是否在用户视线焦点附近（如输入框旁）呈现，而非顶部被忽略的横幅
6. 引用回复的操作是否简便（如选中即弹出引用按钮）
7. 移动端的手势操作是否与系统标准冲突

## 常见坑与对策
1. 控件堆叠导致的认知过载
坑：将十几个插件不加分类地塞入一个菜单，用户无从下手。
对策：按功能维度（如 数据分析、内容创作、辅助工具）进行分组折叠，仅外露高频项。

2. 品牌化命名造成的理解障碍
坑：使用 灵感火花、深思模式 等抽象词汇，用户不敢点击。
对策：回归功能性命名，或在品牌词后由副标题解释具体行为（如 深度思考：进行多步推理）。

3. 状态与行为分离
坑：用户选了 生成表格，但系统回答还是纯文本，导致信任崩塌。
对策：在 System Prompt 中强制注入控件对应的格式约束，或使用 Structured Output 保证执行。

## 可用于丰富《AI 辅助软件产品》的写作点
1. 第 4 章 原型与信息架构
可引用 NN/G 的四类用途（发现、教育、约束、追问）作为 AI 产品原型的设计原则，强调 Hybrid UI 是从 Chatbot 走向 Copilot 的必经之路。

2. 第 6 章 UI 设计
在讨论输入区设计时，建议摒弃单一文本框模式，推荐采用 输入框 + 快捷指令栏 + 侧边配置面板 的组合布局。案例可对比 Perplexity 的 Focus 模式与传统搜索框的区别。

3. 第 8 章 前端实现
建议增加一节 提示词工程的组件化，讲解如何将 System Prompt 的参数（如 temperature, output_format）映射为前端可操作的 Slider 或 Radio Group，实现 把可控性做成界面。

4. 第 11 章 用户模块
在用户引导（Onboarding）部分，提出利用 Prompt Controls 代替传统的全屏遮罩引导。通过预置的 对话启动器 引导用户完成第一次由 UI 辅助的高质量交互，建立即时反馈。
