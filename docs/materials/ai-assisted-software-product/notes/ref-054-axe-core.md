# [54] axe-core：可访问性自动化检查的最低成本方案

- 资料类型：官方文档/工具
- 原始来源：https://github.com/dequelabs/axe-core
- 对应章节：第 3 章（UI/UX）、第 4 章（质量门禁）

## 一句话
可访问性不是“上线前检查一下”，而是“每次改 UI 都别退化”的回归门禁。

## 你该从 axe-core 带走什么
- **自动化能覆盖一部分问题**：能提前发现缺少 label、对比度等常见问题，但不能替代人工走查。
- **更适合 CI**：可以把 a11y 失败当作阻断合并的条件，避免问题积累。
- **输出可追踪**：报告要能定位到具体组件与规则，方便修复。

## 在本书里怎么用
- 第 3 章把它写进“示例（可复制）”的验证命令：a11y 不过就不合并。
- 第 4 章把它纳入质量闸门：减少“只靠审美评审 UI”的争论。

## 常见误用
- 认为跑过一次 axe-core 就等于完全合规，忽略键盘可达与真实任务完成率。
- 不留基线/不留报告，导致每次改动都只能靠感觉。
