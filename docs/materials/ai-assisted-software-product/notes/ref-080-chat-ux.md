# [80] Customer-Service Chat UX（NN/G）：把对话做成可恢复的闭环

- 原始来源：https://www.nngroup.com/articles/chat-ux/
- 对应章节：第 4 章（原型与信息架构）、第 6 章（UI 设计）、第 8 章（前端实现）、第 18 章（评测体系）

## 一句话摘要
客服聊天的体验不是聊天气泡画得像不像，而是入口好不好找、身份清不清楚、等待是否可预期、转人工是否不丢上下文、对话断了能不能回来。

## 你应该从这里带走什么
- 把入口放在用户预期的位置，别只靠悬浮角标；用户找不到入口时，体验再好也没有意义。
- 把身份说清楚，机器人就是机器人，人工就是人工；混在一起会让用户误判能力边界。
- 把等待与状态写成可预期信息，尤其在移动端，缺反馈会直接触发退出。
- 把跨渠道衔接当作强约束：转人工后重复问一遍已提供信息，会立刻摧毁信任。
- 把对话记录当作证据与资产：可回看、可导出、可追溯，能显著降低争议与支持成本。

## 落地要点（可执行）
- 入口策略：在联系我们等页面放明确入口，并允许从具体页面带上下文发起对话。
- 身份与能力边界：欢迎语直接说明当前是机器人还是人工，以及下一步能做什么。
- 上下文继承：转人工时透传摘要与关键字段，避免重复收集信息。
- 会话保持：刷新、弱网、切后台后能恢复对话，不重新排队，不丢历史。
- 体验评测：把首响时间、转人工后重复提问率、移动端中断恢复成功率写成回归指标。

## 常见误用提醒
- 只优化话术不优化机制：再礼貌的欢迎语也救不了丢上下文、断线重排队。
- 把聊天当作万能入口：客服聊天适合解决问题，不适合承载复杂产品教育，别把所有引导都塞进对话里。
