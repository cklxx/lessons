# [74] RFC 2119：把“必须/应该/可以”写成团队一致的合同语言

- 原始来源：https://www.rfc-editor.org/rfc/rfc2119
- 对应章节建议：第 3 章（PRD 与工程合同）+ 第 7 章（工程化与门禁）

## 一句话摘要
RFC 2119 给“需求文档里那些含糊的口气”一个严格语义：**MUST/SHOULD/MAY** 不是情绪，而是契约；它让你能把 PRD 写得更像规格书、也更像验收条款。

## 你应该从这里带走什么（面向产品）
- **把模糊口吻变成可裁决条款**：当你写“应该尽量快”“最好有提示”，团队无法验收；把它改写成 MUST/SHOULD + 明确边界与例外。
- **把例外写进合同**：哪些情况允许降级？哪些情况必须拒绝服务？在 AI 产品里，这些是体验与成本/风险的交界线。

## 写作落地建议（可直接套用）
- 在 PRD 里建立一个小节：`Terminology`，声明本文使用 RFC 2119 语义。
- 把每个关键模块的验收条款写成“规范句”：
  - “当预算达到阈值时，系统 MUST 降级或拒绝，并向用户解释原因与自救方式。”
  - “当输出含外部事实时，系统 SHOULD 提供来源或标记‘待核验’。”

## 常见误用提醒
- **把 MUST 写满整篇**：MUST 代表强约束；如果一切都是 MUST，实际上就等于没有优先级。
